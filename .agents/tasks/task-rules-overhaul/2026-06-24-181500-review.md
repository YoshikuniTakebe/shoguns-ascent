# Complete board game rules overhaul for Shogun's Ascent

This change replaces the simplified game prototype with a full implementation of the board game rules: 8 clans with proper figure types (Daimyo, Bushi, Shinto, Fortress), 8 provinces connected by land adjacency and sea routes, a temple/kami system, 85+ season cards across three decks, war tactics bidding, hostages, and a six-phase seasonal loop (Setup, Tea, Politics, War, Cleanup, Winter). The implementation totals ~1270 lines of game logic, a Zustand store with 20+ actions, a WebSocket server with matching message handlers, and 11 UI components.

**Watch for:** Marshal mandate is a no-op (confirmed) — it logs a message but performs no movement. Coin spending is never validated against player balance in war tactic bids (confirmed). Cleanup discards all coins unconditionally, including those that might be needed for winter scoring cards that reference coin-based conditions (confirmed). The `doChooseMandateTile` store action calls both `chooseMandateTile` (which internally calls `executeMandate`) and then `advancePlayer`, meaning mandate choice and execution are atomic with the turn advance — but the separate `doExecuteCurrentMandate` action exposes a second path that executes a mandate without advancing, creating a potential double-execute (likely).

## High-level view

The data model introduces a province-based map of 8 regions (Hokkaido through Kyushu) with explicit adjacency and sea-route graphs, replacing the old generic "regions with force counts" approach. Figures are individual entities with IDs, types, and ownership rather than integer force counters, which enables the war tactics and hostage mechanics. The honor track is a positional array where higher index means higher honor, used as tiebreaker throughout battle and harvest resolution.

The politics phase implements a 7-mandate structure with interleaved kami turns (3+kami+2+kami+2+kami). Mandate resolution follows a left-of-issuer order with an issuer/ally bonus pattern. The critical gap is that Marshal is stubbed: it logs but does not programmatically move any figures, relying entirely on the player to use the separate `moveForces` function through the UI. Since other mandates auto-execute for all players in resolution order (Recruit places bushi, Harvest pays coins, Betray replaces figures), Marshal breaks the symmetry — non-issuer players never get their mandated move resolved.

War resolves through a tactics-bidding system (Seppuku, Take Hostage, Hire Ronin, Imperial Poets) followed by force comparison. The bid coins are never deducted from the bidder's balance at submission time; only losers have their coins deducted post-battle and the winner's bid is redistributed to losers. This means a player can bid coins they don't have.

Season cards exist as data definitions (85+ cards) but the Train mandate only gives a bonus coin to the issuer/ally. No card is actually purchased or added to a player's `seasonCards` array during the Train flow — the purchase would need to happen through a UI action that doesn't exist yet.

## Marshal and Train: incomplete mandate implementations

`executeMarshal` is a log-only function:

```typescript
function executeMarshal(state: GameState, issuerId: string): GameState {
  const newState: GameState = { ...state, log: [...state.log, `Marshal mandate issued by ${state.players.find((p) => p.id === issuerId)?.name} - players may move figures`] };
  return newState;
}
```

Every other mandate (Recruit, Harvest, Betray) iterates `getResolutionOrder` and performs the action for each player automatically. Marshal breaks this pattern. In hotseat mode, only the current player can use `toggleMoveMode` to move forces — other players never get a UI prompt for their mandated move. The ActionPanel shows a "Move Forces" button only for the active player during the politics phase, not for each player in resolution order.

`executeTrain` similarly gives a bonus coin to issuer/ally but doesn't implement the actual card-purchase mechanic. The `SeasonCardsMarket` component renders cards but has no buy button or store action for purchasing. The 85+ season card definitions are effectively dead data until this flow is built.

## Coin economy: unvalidated bidding and premature zeroing

War tactic bids flow through `submitWarTacticBids` which accepts arbitrary bid amounts:

```typescript
export function submitWarTacticBids(state, provinceId, bids): GameState {
  const battle = newState.activeBattles.find((b) => b.provinceId === provinceId);
  battle.warTacticBids = bids;
  return newState;
}
```

No check ensures `sum(bids[playerId]) <= player.coins`. The battle resolution later deducts coins from losers with `Math.max(0, loser.coins - totalBid)`, which clamps negative values but allows the overbid to succeed. The winner's coins are never deducted — they're distributed to losers instead. This creates an asymmetry: the winner effectively spends nothing (their bid goes to losers), while losers lose their bid to the void.

Separately, `cleanupSeason` sets `coins: 0` for all players unconditionally. Since cleanup happens before the next season's `setupSeason` distributes income, this means any coins accumulated during war (e.g., from winning Hire Ronin or Take Hostage) are immediately lost. The real game rules discard coins at cleanup, so this may be intentional — but it means the `coins` field is almost always 0 at season boundaries.

## Dual mandate-execution paths in the store

The store exposes two actions that execute mandates:

1. `doChooseMandateTile(mandate)` → calls `chooseMandateTile(state, mandate, apid)` which internally calls `executeMandate`, then calls `advancePlayer` on the result.
2. `doExecuteCurrentMandate(mandate)` → calls `executeMandate(state, mandate, apid)` directly.

The ActionPanel renders both: a mandate-choice button (calling `doChooseMandateTile`) *and* a "Current Mandate" section with an "Execute Mandate" button (calling `doExecuteCurrentMandate`). If a player clicks the mandate tile and then clicks "Execute Mandate" on the resulting `mandatesThisTurn` entry, the mandate fires twice. The `mandatesThisTurn` array grows by one each time `executeMandate` is called, so the second click would append a duplicate entry and re-execute the mandate effects (double recruitment, double harvest, etc.).

## Hire Ronin: tied-bid bug in battle resolution

In `resolveNextBattle`, after war tactics are resolved, the ronin force bonus is applied:

```typescript
const hireRoninBid = battle.warTacticBids[pid]?.['hire-ronin'] || 0;
if (hireRoninBid > 0) {
  const isHighest = battle.participants.every((otherId) => {
    if (otherId === pid) return true;
    return (battle.warTacticBids[otherId]?.['hire-ronin'] || 0) <= hireRoninBid;
  });
  if (isHighest) {
    force += player.ronin;
  }
}
```

The `<=` comparison means that if two players both bid 3 on hire-ronin, both pass the `isHighest` check and both get their ronin force added. The earlier tactic-resolution loop correctly uses honor as a tiebreaker for determining which player *wins* the hire-ronin slot, but this later force-calculation block doesn't respect that resolution — it independently re-evaluates who "won" using a different (broken) criterion.

## Server message-type collapse

Six distinct client-side actions all emit `{ type: 'ADVANCE_PHASE' }` in online mode: setup season, resolve kami, initiate war, resolve next battle, cleanup season, and resolve winter. The server handler:

```typescript
case 'ADVANCE_PHASE': {
  l.gameState = advancePhase(l.gameState);
  broadcastState(l);
  break;
}
```

`advancePhase` switches on `currentPhase` and performs a single predetermined transition. This works if the client always sends ADVANCE_PHASE at the right moment, but it means the server cannot perform `resolveNextBattle` or `initiateWarPhase` directly — it only advances to the next phase. A client sending ADVANCE_PHASE during the war phase will transition to cleanup, even if battles remain unresolved.

## Winter scoring: Form of the Kindred and alliance history

`scoreWinterUpgrade` for `au-form-of-the-kindred` returns `player.allies.length > 0 ? 9 : 0`. The card's text says "3 VP per Season you had an ally" — meaning a player who was allied in only 1 of 3 seasons should score 3 VP, not 9. Since there's no alliance-history tracking (alliances are broken and reformed each season, and the only state is the current `allies` array), this scoring always awards the maximum if the player happens to have an ally during winter resolution.

## Province graph and map symmetry

The 8 provinces form a linear chain with a few cross-connections and sea routes:

```
Hokkaido -- Oshu -- Edo -- Kanto
                     |       |
                    Kansai --+
                   /     \
              Nagato    Shikoku
                |
              Kyushu
Sea: Hokkaido<->Shikoku, Nagato<->Shikoku, Shikoku<->Kyushu
```

Edo has 3 land connections and a central position, making it the strongest province for harvest (reward 4) and for military pressure. Hokkaido and Kyushu are peripheral (1 land connection each, 1 sea route), which disadvantages the Koi and Luna clans positioned there. This asymmetry may be intentional for the board game's balance, but it's worth noting that the clan-to-province assignment gives some clans structurally stronger starting positions.
