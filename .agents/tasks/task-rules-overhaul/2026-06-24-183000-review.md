# Complete board game rules overhaul (v2)

All 8 v1 issues have been addressed. Marshal has improved logging with bonus-move documentation, `buySeasonCard` implements a full purchase flow with cost validation, war tactic bids are validated against coin balance, the duplicate `doExecuteCurrentMandate` path is removed, cleanup coin discard is confirmed correct per rules, `allianceSeasons` tracking supports Form of the Kindred scoring, Hire Ronin tied-bid uses a pre-computed `hireRoninWinner` with honor tiebreaker, and the server handles 12 distinct message types.

**Watch for:** In online multiplayer, `SUBMIT_WAR_BIDS` auto-resolves the battle immediately after one player's bids are stored, meaning other participants never get to bid (confirmed). The `submitWarTacticBids` function replaces `battle.warTacticBids` entirely rather than merging, so a second player's submission overwrites the first's (confirmed). The `doCleanupSeason` store action and the `advancePhase` 'cleanup' case both call `cleanupSeason`, creating a potential double-cleanup if both are triggered in sequence (likely).

**Verdict**: NEEDS_CHANGES

## High-level view

The server message-type expansion fixes the v1 ADVANCE_PHASE collapse but introduces a new issue: the `SUBMIT_WAR_BIDS` handler calls `resolveNextBattle` immediately after storing bids. In online mode, the first player to submit triggers resolution before other combatants have bid, making online multiplayer war tactics meaningless.

A lifecycle gap exists in the cleanup path: `doCleanupSeason` (button) and the 'cleanup' case in `advancePhase` (reachable via `doAdvancePhase`) both invoke `cleanupSeason`. The UI prevents double-firing through conditional rendering, but the server's `CLEANUP_SEASON` followed by `ADVANCE_PHASE` would double-clean — zeroing coins distributed by `setupSeason` and double-incrementing `allianceSeasons`.

Season card purchases are unrestricted during the entire politics phase. Per rules, buying should only be available during/after Train mandate resolution, but no flag or sub-phase gates the `buySeasonCard` call.

<summary>Issues (4)</summary>

1. **Online battle auto-resolve on first bid** — Server `SUBMIT_WAR_BIDS` handler calls `resolveNextBattle` immediately after `submitWarTacticBids`, resolving the battle before other participants submit their bids. Remove the auto-resolve or gate it on all participants having submitted.
2. **War tactic bids overwrite instead of merge** — `submitWarTacticBids` sets `battle.warTacticBids = bids`, replacing all existing bids. A second submission discards earlier players' bids. Merge incoming bids into existing ones.
3. **Double cleanup path** — `doAdvancePhase` at the 'cleanup' case calls `cleanupSeason` internally before advancing to next season. If the server receives `CLEANUP_SEASON` then `ADVANCE_PHASE`, or if the UI exposes both paths, cleanup runs twice — double-incrementing `allianceSeasons` and zeroing freshly-distributed coins. Either remove `doCleanupSeason` or make the cleanup case in `advancePhase` idempotent.
4. **Season card purchase unrestricted to Train mandate** — `buySeasonCard` can be called at any time during politics phase. Per rules, card purchase should only be available during/after Train mandate resolution. Add a guard flag set by `executeTrain` and cleared on next mandate.

## Online war bid resolution is premature

The server handler for `SUBMIT_WAR_BIDS`:

```typescript
case 'SUBMIT_WAR_BIDS': {
  const l = lobbies.get(currentLobbyId || '');
  if (!l?.gameState) return;
  const { provinceId, bids } = data.payload || {};
  if (!provinceId || !bids) return;
  l.gameState = submitWarTacticBids(l.gameState, provinceId, bids);
  // Auto-resolve battle after bids are submitted
  l.gameState = resolveNextBattle(l.gameState);
  broadcastState(l);
  break;
}
```

In a 3-player battle, the first player to submit triggers `resolveNextBattle`. The other two players' bids never make it into `battle.warTacticBids` because the battle is already resolved.

The root cause is compounded by `submitWarTacticBids` replacing rather than merging: `battle.warTacticBids = bids`. Even without the auto-resolve, a second player's submission overwrites the first. The fix requires both accumulation (merge bids into the existing map keyed by playerId) and a gate (only call `resolveNextBattle` once all `battle.participants` have entries in `warTacticBids`).

## Double-cleanup lifecycle gap

Two paths lead to cleanup execution:

1. ActionPanel "End War Phase" → `doAdvancePhase()` → `advancePhase` transitions to `'cleanup'` phase (sets phase only, no cleanup logic runs)
2. ActionPanel "Proceed to Next Season" → `doCleanupSeason()` → `cleanupSeason()` runs the actual cleanup

But `advancePhase` at the 'cleanup' case also calls `cleanupSeason()` before advancing to the next season. The server exposes both `CLEANUP_SEASON` (calls `cleanupSeason`) and `ADVANCE_PHASE` (calls `advancePhase` which, if phase is 'cleanup', calls `cleanupSeason` again). A malicious or buggy client sending both messages — or a race between them — would double-increment `allianceSeasons` for allied players and zero out coins that `setupSeason` just distributed.

## Season card purchase timing

```typescript
const canBuy = gameState.currentPhase === 'politics';
```

The SeasonCardsMarket enables purchase for the entire politics phase. A player can buy cards on mandate turn 1 (before any Train is issued), spend coins that should be available for war bidding, or buy multiple cards across different Train mandates without restriction. The mechanical impact: the coin economy during war is undermined because players can pre-spend coins on cards at any time.
