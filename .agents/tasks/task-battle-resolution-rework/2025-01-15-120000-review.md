# Step-by-step battle resolution with Seppuku, Hostage, Ronin, and Imperial Poets popups

This change replaces the one-shot battle resolution with an interactive step-by-step popup flow. After all tactic bids are submitted, the game walks through each tactic winner in sequence: Seppuku decision/result, Hostage selection, Ronin announcement, then final force comparison. It also fixes fortress-only battle exclusion (non-Tortuga fortresses no longer drag players into battles), hides exhausted mandates from the PoliticsTrack, corrects the Sol clan log message format, and makes the Koi clan's ronin counter in the bidding overlay decrease dynamically as coins are placed.

**Watch for:** The Koi ronin display shows `maxCoins - totalAssigned` which represents unspent coins, but Koi's actual ronin strength in combat is `player.ronin + remaining coins after ALL bids` — these are different values and the display may mislead the player (confirmed). The `take-hostage` fallback path (non-step-by-step) doesn't allow capturing monsters, while the step-by-step popup does — an asymmetry in hostage-eligible figures between the two code paths (confirmed). Phoenix double-death tracking for Imperial Poets relies on the seppuku owner losing the battle, but doesn't verify that the specific Phoenix figure was actually killed vs surviving as a captured hostage (likely).

## High-level view

The battle state machine in the store grows from three states (`popup`, `bidding`, `result`) to seven, adding `seppuku-decision`, `seppuku-result`, `hostage-selection`, and `ronin-result`. A new `determineTacticWinners` function pre-computes who won each tactic from the bids without applying effects, then the store drives popups for each phase in sequence. The `resolveNextBattle` function in gameLogic detects whether it was called from the step-by-step flow (via `resolutionData` on the battle) and skips seppuku/hostage resolution if the store already handled them interactively.

Seppuku now kills ALL troop types (bushi, shinto, daimyo, monster), fixing the previous omission of shinto. Phoenix revival is implemented in both paths: Phoenix revives immediately after seppuku, making it available for hostage capture or a second death in the final resolution. The fallback `take-hostage` path only captures bushi/shinto while the interactive path also allows monster capture — this gap should be unified.

The Koi ronin display during bidding shows remaining pool coins, but the actual combat formula includes base ronin tokens plus remaining coins after all bids are finalized. The number the player sees during bidding doesn't match what will be applied in combat.

## Koi ronin display formula

In `BattleBiddingOverlay.tsx`, the Koi ronin count is displayed as:

```tsx
playerClanId === 'koi' ? Math.max(0, maxCoins - totalAssigned) : playerRonin
```

`totalAssigned` is `maxCoins - poolCoins.length` — the number of coins dragged from the pool to any tactic slot. So the display shows how many coins remain in the pool. But Koi's actual ronin force during final battle calculation (in `resolveNextBattle`) is:

```ts
force += player.ronin;
const totalBidByPlayer = Object.values(battle.warTacticBids[pid] || {}).reduce((s, v) => s + v, 0);
const remainingCoins = player.coins - totalBidByPlayer;
force += Math.max(0, remainingCoins);
```

This adds `player.ronin` (base ronin tokens) plus remaining coins after final bids. The UI only shows the coin portion, not base ronin, and during bidding the "remaining coins" may differ from "coins in pool" if the bidding overlay's `maxCoins` doesn't equal `player.coins`. The player sees a number that looks like total ronin strength but is actually just unplaced coins.

## Hostage-eligible figure asymmetry between paths

The step-by-step hostage selection in `BattlePanel.tsx` filters capturable figures:

```tsx
const capturableFigures = province.figures.filter(f => {
  if (f.owner === battleResolutionData.hostageWinnerId) return false;
  if (f.type === 'daimyo') return false;
  if (f.type === 'fortress') return false;
  if (f.type === 'monster' && f.monsterCardId && DAIMYO_MONSTER_IDS.includes(f.monsterCardId)) return false;
  if (f.type === 'bushi' || f.type === 'shinto' || f.type === 'monster') return true;
  return false;
});
```

Monsters (non-daimyo-type) are capturable here. But the fallback path in `resolveNextBattle`:

```ts
const enemyFigure = curProv.figures.find(
  (f) => f.owner !== highestBidder && (f.type === 'bushi' || f.type === 'shinto')
);
```

Only bushi/shinto. If the non-step-by-step path ever runs (online mode without store-driven popups), monsters can't be captured.

## Phoenix double-death tracking gaps

The `phoenixDiedInBattle` flag in `resolveNextBattle` (step-by-step mode):

```ts
if (stepByStepMode && resData.phoenixDiedInSeppuku) {
  const seppukuOwnerId = resData.seppukuWinnerId;
  if (seppukuOwnerId && seppukuOwnerId !== winnerId && !winner.allies.includes(seppukuOwnerId)) {
    phoenixDiedInBattle = true;
  }
}
```

This infers Phoenix died in battle because the seppuku player lost. But if the revived Phoenix was captured as a hostage (already removed from province by `doHostageConfirm`), it wouldn't die in battle resolution. The code would still set `phoenixDiedInBattle = true`.

The fix: check `resData.capturedHostage` — if it references the Phoenix (`monsterCardId === 'sp-phoenix'`), skip the `phoenixDiedInBattle` flag. Currently `capturedHostage.figureType` stores the type but not the `monsterCardId`, so the data needed for this check isn't fully available on the resolution data.

## Hostage death counting inconsistency between paths

In the fallback (non-step-by-step) take-hostage resolution:

```ts
battleDeathCount += 1;
```

The hostage is counted as a death for Imperial Poets. But in step-by-step mode, hostage capture happens in the store before `resolveNextBattle` is called, and `battleDeathCount` starts at `resData.seppukuKillCount` without any hostage contribution. This means Imperial Poets awards differ depending on which code path resolves the battle — one VP more in the fallback path for the same game situation.
