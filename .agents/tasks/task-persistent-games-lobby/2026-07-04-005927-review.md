# Persistent games lobby and replay viewer (v3)

All blocking issues from v1 and v2 are resolved: the API contract mismatch in `gameStore.ts` (v1) and the `GamesLobby` field names (v2) have been fixed. The lobby now uses `players`, `createdAt`, `updatedAt` from the formatted API response. The store correctly reads `data.id` for hotseat persistence, uses pre-parsed `state` objects for resume and replay, and the build passes cleanly.

**Watch for:** The kami-jump button is still non-functional — it scans snapshot descriptions for the substring "kami", but the server generates descriptions as `"politics - spring (round 1)"` which never contains that word (confirmed). The battle-jump works correctly because it matches on `phase === 'war'`. The unbounded snapshot memory concern remains a design limitation but is not a shipping blocker for an initial release.

**Verdict**: APPROVED

## High-level view

The kami-jump navigation remains dead UI. The `replayNextKami`/`replayPrevKami` functions scan for `description.toLowerCase().includes('kami')`, but the PUT `/api/games/:id/snapshot` endpoint constructs descriptions as `"${state.currentPhase} - ${state.currentSeason} (round ${state.round})"` — the word "kami" never appears. The battle-jump works because it matches on the `phase` metadata field (`'war'`), which is correctly stored from `state.currentPhase`. This is a minor UX gap, not a correctness issue.

The full-snapshot-in-memory design loads all snapshots in one fetch. For a typical hotseat game (20-60 snapshots), this is tolerable. For online games, where `broadcastState` saves on every WebSocket action, snapshot counts can reach 200-500 per game.

## Kami-jump description mismatch

The kami-jump logic in `replayNextKami`:

```typescript
if (replaySnapshots[i].phase === 'politics' && replaySnapshots[i].description.toLowerCase().includes('kami')) {
  set({ replayCurrentIndex: i });
  return;
}
```

The server's PUT `/api/games/:id/snapshot` handler constructs descriptions as:

```typescript
const description = `${state.currentPhase} - ${state.currentSeason} (round ${state.round})`;
```

A kami-resolution snapshot gets description `"politics - spring (round 1)"`. The `includes('kami')` check always fails. The button appears enabled (disabled logic only checks boundary indices) but clicking it does nothing.

The online path has the same issue — `broadcastState` uses the identical description template.

Fix options: (a) include "kami" in the description when `state.kamiResolutionActive` is true at the time of `doResolveKami`'s `saveSnapshot` call (but this runs post-resolution so the flag may be cleared), (b) change the scan predicate to use a separate metadata field, or (c) accept the limitation and document it. Since the battle-jump works via `phase === 'war'` matching, the kami-jump could similarly work if kami resolution used a distinct phase value, but it doesn't — it stays in `'politics'`.

## Online snapshot density

Every `broadcastState` call saves a snapshot. In the online WebSocket handlers, DRAW_MANDATE_TILES, CHOOSE_MANDATE, MOVE_FIGURE, SUBMIT_WAR_BIDS, FORM_ALLIANCE, BREAK_ALLIANCE, ADVANCE_PHASE, SETUP_SEASON, RESOLVE_KAMI, INITIATE_WAR, RESOLVE_BATTLE, RESOLVE_WINTER, PLACE_FIGURE, BUY_CARD, SKIP_MARSHAL_TURN, BUILD_FORTRESS, BETRAY_SELECT_FIGURE, and SKIP_BETRAY_TURN all produce snapshots. A 4-player game could easily reach 200-500 snapshots. The replay fetch loads all of them with full `GameState` in a single response.
