# Persistent games lobby and replay viewer (v2)

The critical API contract mismatches from v1 have been fixed in `gameStore.ts`: `loadReplayGame` now reads `snapshotIndex` and uses `state` directly, `resumeGame` uses the pre-parsed state object, and `createGame` correctly reads `data.id` from the hotseat save response. A `saveSnapshot` call was added after `doLotoChooseActualMandate` to improve replay coverage.

**Watch for:** The `GamesLobby` component still uses the raw DB column names (`players_json`, `created_at`, `updated_at`) in its local `GameRecord` interface, while the API returns camelCase (`players`, `createdAt`, `updatedAt`) (confirmed). This means lobby cards render with no player names and no dates. The full-snapshot-in-memory design remains — all snapshots are loaded in a single fetch and held in the Zustand store, producing unbounded memory growth for long games (confirmed). Replay jump buttons (kami/battle) silently no-op when no matching snapshot exists (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The same class of contract bug fixed in `gameStore.ts` persists in `GamesLobby.tsx`: the component defines a `GameRecord` interface with `players_json`, `created_at`, `updated_at` and calls `JSON.parse(game.players_json || '[]')`, but the API response contains `players` (already an array), `createdAt`, and `updatedAt`. The lobby renders cards without player names and without dates.

Snapshot coverage improved with `doLotoChooseActualMandate` now calling `saveSnapshot`, but recruit, marshal, build fortress, and betray actions remain un-snapshotted. Replay jumps from mandate selection to phase-advance with no intermediate visibility.

The kami-jump and battle-jump controls scan the snapshot array for state predicates that may never be true in any stored snapshot. `doResolveKamiTurn` snapshots *after* resolution (when `kamiResolutionActive` is likely cleared), and `doResolveBattle` snapshots after the battle is removed from `activeBattles`. The jump buttons become dead UI.

## GamesLobby API contract mismatch

The fix applied in FEAT-003 corrected `loadReplayGame`, `resumeGame`, and the hotseat `createGame` handler. `GamesLobby.tsx` was not updated. Its `GameRecord` interface:

```typescript
interface GameRecord {
  id: string;
  name: string;
  players_json: string;    // API sends "players" (already parsed)
  status: string;
  created_at: string;      // API sends "createdAt"
  updated_at: string;      // API sends "updatedAt"
  mode: string;
  winner: string | null;
}
```

The render logic calls `JSON.parse(game.players_json || '[]')` — since `game.players_json` is `undefined`, this evaluates to `JSON.parse('[]')` producing an empty player list. The date display reads `game.updated_at || game.created_at`, both `undefined`, so `formatDate` receives `undefined` and the catch block returns the string "undefined".

The fix: change the interface to match the API (`players: { name: string; clanId: string }[]`, `createdAt: string`, `updatedAt: string`) and use `game.players` directly.

## Replay memory footprint

Each snapshot in `replaySnapshots` is a fully-parsed `GameState` — provinces with figure arrays, players with card arrays, `activeBattles`, `log` (growing string array). Conservative estimate: 50-100 KB per snapshot.

A completed 4-player game across 3 seasons with snapshot calls on phase-advance, kami, battle, winter, and mandate choice produces 60-200 snapshots. Upper bound: 200 × 100KB = 20 MB in-browser, problematic on mobile.

The `getSnapshotByIndex` endpoint already exists. A fetch-on-demand approach (load metadata list first, fetch full state per index as the user navigates) would keep memory bounded to one `GameState` at a time.

## Kami/battle jump predicates never match

The kami-jump scans for `snapshots[i].state.kamiResolutionActive === true`. The battle-jump scans for `state.currentPhase === 'war' && state.activeBattles.length > 0`.

`doResolveKamiTurn` calls `resolveKamiTurn(gameState)` which processes and completes the kami turn, then snapshots the result. If `resolveKamiTurn` clears `kamiResolutionActive` as part of resolution, every kami snapshot has `kamiResolutionActive === false`. The jump never finds a match.

`doResolveBattle` calls `resolveNextBattle` which resolves a battle and removes it from `activeBattles`, then snapshots. When the last battle in a province is resolved, the snapshot has `activeBattles.length === 0`. The battle-jump scan misses it.

Both buttons' disabled-state logic only checks boundary indices (first/last), not whether a target exists. They appear enabled but do nothing when clicked.

Fix: snapshot *before* resolution to capture the active state, or change the scan predicates to `phase === 'kami'` and `phase === 'war'` (using the snapshot metadata `phase` field rather than inspecting `GameState` internals).
