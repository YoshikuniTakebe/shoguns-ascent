# SQLite persistence, games lobby, and replay viewer for Shogun's Ascent

This change adds a SQLite-backed persistence layer to the game server, a frontend games lobby for browsing active/finished games, and a step-by-step replay viewer with kami/battle jump controls. The backend stores game metadata and full-state snapshots via better-sqlite3, exposing REST endpoints for CRUD. The frontend integrates into the existing Zustand store with fire-and-forget persistence calls from hotseat mode, and fetches snapshot history for replay navigation.

**Watch for:** A client/server API contract mismatch breaks both replay loading and game resume (confirmed) ‚Äî the client expects raw DB column names (`state_json`, `snapshot_index`, `players_json`) but the server returns camelCase transformed responses with already-parsed objects. Hotseat persistence is silently non-functional because the response property name mismatch prevents `persistentGameId` from ever being set (confirmed). Loading all snapshots (full game state each) in a single fetch for replay will produce multi-megabyte payloads for any non-trivial game (likely).

## High-level view

The persistence layer stores the complete game state per snapshot with no delta compression, which becomes a storage and bandwidth concern as games grow.

The client-server contract is broken in multiple places. The server's `formatGame` helper and snapshot endpoints return camelCase keys with parsed JSON, but the client's `loadReplayGame` and `resumeGame` functions destructure raw DB column names and call `JSON.parse` on already-parsed objects. The hotseat save endpoint returns `{ id }` but the client reads `data.gameId`, so `persistentGameId` is never set and subsequent `saveSnapshot` calls no-op.

The replay viewer provides the requested navigation controls (step, kami-jump, battle-jump) via linear scans over the in-memory snapshot array. It displays player stats and province summaries but does not reuse the game board, limiting replay's visual utility.

Snapshot calls are placed after key state transitions (phase advance, battle resolution, kami resolution, winter), but game actions like mandate choices, recruit placements, and marshal moves don't trigger snapshots, producing gaps in the replay timeline.

1. **Snapshot field name mismatch in `loadReplayGame`** ‚Äî The server returns `{ snapshotIndex, state }` but the client expects `{ snapshot_index, state_json }` and calls `JSON.parse` on the already-parsed state object. Fix the client to use the API's camelCase response shape.
2. **`players_json` reference in `loadReplayGame`** ‚Äî The API returns `players` (parsed array) but the client reads `game.players_json`, which is `undefined`. This silently produces an empty player list in replay metadata.
3. **Hotseat `persistentGameId` never set** ‚Äî Server returns `{ id: gameId }` but client checks `data.gameId`. Since `persistentGameId` stays null, all subsequent `saveSnapshot()` calls bail out immediately. Hotseat games are saved once on creation but never get additional snapshots.
4. **`resumeGame` calls `JSON.parse` on parsed object** ‚Äî Same API mismatch as replay: `lastSnapshot.state_json` is `undefined` (API returns `state` already parsed). Resume will throw or produce garbage state.
5. **Full-state snapshot payload for replay** ‚Äî `/api/games/:id/snapshots` returns all snapshots with full parsed game state in a single response. A 200-move game could produce 5-10 MB of JSON. Consider paginated loading or on-demand fetching per index.
6. **Incomplete snapshot coverage** ‚Äî `saveSnapshot()` is called after `advancePhase`, `resolveKamiTurn`, battle resolution, and winter, but not after mandate choices, recruit placements, or marshal actions. Replay will skip these transitions.

## Client-server API contract mismatch

The server's REST layer transforms database rows before sending them. `formatGame` converts `players_json` (string) into `players` (parsed array) and renames `created_at`/`updated_at` to `createdAt`/`updatedAt`. The snapshots endpoint parses `state_json` into `state` and renames `snapshot_index` to `snapshotIndex`.

The client code in `loadReplayGame` was written against the raw DB schema:

```typescript
const parsedSnapshots = snapshots.map((s: { snapshot_index: number; state_json: string; ... }) => ({
  index: s.snapshot_index,       // undefined ‚Äî API sends snapshotIndex
  state: JSON.parse(s.state_json) as GameState, // JSON.parse on object ‚Üí crash or "[object Object]"
  ...
}));
const players = JSON.parse(game.players_json || '[]'); // game.players_json is undefined
```

The same pattern appears in `resumeGame`:

```typescript
const gameState = JSON.parse(lastSnapshot.state_json) as GameState; // state_json is undefined
```

Both replay and resume are non-functional as written. The fix: use the API's response shape (`s.snapshotIndex`, `s.state`, `game.players`).

## Hotseat persistence silently disabled

In `createGame`, after calling `/api/games/save-hotseat`, the response handling reads:

```typescript
.then(data => {
  if (data && data.gameId) {     // server returns { id: "..." }
    set({ persistentGameId: data.gameId });
  }
})
```

Since `data.gameId` is always `undefined`, `persistentGameId` remains null for the entire session. No snapshots are ever appended after game creation. Resuming a hotseat game always reloads the starting state, and replay shows a single frame.

## Snapshot volume and memory pressure

`loadReplayGame` fetches all snapshots in one request and stores fully-parsed `GameState` objects in the Zustand store's `replaySnapshots` array. A `GameState` includes provinces with all figures, player data, card arrays, honor tracks, and battle state. For a full 4-player game lasting 3 seasons with 100-300 snapshots at 50-100 KB each, expect a 5-30 MB fetch response and equivalent in-browser memory consumption.

A paginated approach ‚Äî fetching snapshot-count first, then loading states on demand per index ‚Äî would keep memory bounded.

## Snapshot placement gaps

Snapshots are triggered by `get().saveSnapshot()` calls in:

- `doAdvancePhase` and `doAdvancePhaseManual`
- `doResolveKamiTurn`
- `doResolveBattle`
- `doResolveWinter`

Missing: `doChooseMandate` (the core political action each turn), `doRecruitPlaceFigure`, `doBuildFortress`, `doMarshalMove`, and `doBetraySelectFigure`. Replay will jump from "start of politics" to "phase advanced" with no visibility into what each player chose or built.

## Kami and battle jump may find no targets

The kami-jump scans for `snapshots[i].state.kamiResolutionActive === true` and the battle-jump for `state.currentPhase === 'war' && state.activeBattles.length > 0`. Since kami resolution snapshots only fire in `doResolveKamiTurn` (which resolves the kami and may clear `kamiResolutionActive` before the snapshot is taken), the jump might never find a matching snapshot and the buttons would silently do nothing.

## Asymmetric snapshot density between modes

For online games, `broadcastState` saves a snapshot on every state broadcast ‚Äî every WebSocket handler that mutates state produces one, including moving a single figure or placing a bid. Online games will accumulate far more snapshots than hotseat, creating inconsistent replay granularity depending on game mode.
