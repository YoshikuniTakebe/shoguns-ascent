# Drag-and-drop battle bidding overlay

Replaces the slider-based bidding UI with a full-screen visual overlay where players drag coin images onto tactic tiles. The new `BattleBiddingOverlay` component renders four tactic tiles (Seppuku, Take Hostage, Hire Ronin, Imperial Poets) with kanji art, a coin pool showing one draggable coin per available coin, and a confirm button. Both hotseat and online battle modes now route through this overlay instead of inline sliders.

**Watch for:** Mobile/touch devices cannot use this interface at all (confirmed — HTML5 drag-and-drop has no touch support). The online-mode overlay renders inside a positioned container rather than as a true full-screen takeover, creating a layout mismatch between modes (confirmed). No enforcement prevents confirming with zero coins allocated (confirmed).

## High-level view

The overlay uses HTML5 drag-and-drop exclusively, which works on desktop browsers but does not fire on mobile touch devices — the entire bidding flow is inaccessible on phones and tablets without a polyfill or touch-event fallback.

In hotseat mode the overlay replaces the entire `BattlePanel` return, achieving a true full-screen modal. In online mode it renders *inside* the existing battle panel `<div>`, so the fixed-position overlay stacks on top of the combatants list in a way that may produce z-index or scroll conflicts rather than a clean modal takeover.

The confirm button submits whatever distribution the player has set, including the case where all coins remain in the pool (zero bid across all tactics). The visual "drag coins to commit" metaphor may mislead players into thinking unassigned coins are withheld rather than wagered at zero.

The combatants info section (force display, bid status per player) that was previously visible during bidding in hotseat mode has been entirely removed. Players no longer see force counts or opponent bid status before committing.

<summary>Issues (6)</summary>

1. **No mobile/touch support** — HTML5 drag-and-drop doesn't fire on touch devices. Add touch event handlers (`onTouchStart`/`onTouchMove`/`onTouchEnd`) or use a polyfill like `react-dnd-touch-backend`.
2. **Online-mode overlay nesting** — The overlay renders inside `.battle-panel` rather than at portal/root level, so `position: fixed` may clip or scroll unexpectedly in some layouts. Render it via a React portal or move it outside the panel wrapper.
3. **Zero-bid confirmation** — Players can confirm with all coins still in the pool. Consider disabling the confirm button or showing a warning when `totalAssigned === 0`.
4. **Lost combatant context in hotseat** — The force display and opponent bid-status indicators were removed from the hotseat bidding view. Players have no information about relative army strength before committing bids.
5. **Stale coin count on remount** — `coinPositions` is initialized once via `useState` initializer. If `maxCoins` changes while the component is mounted (unlikely but possible in edge cases), the pool won't update.
6. **No drop-target highlight** — Tactic tiles have no visual feedback (`dragenter`/`dragleave` class toggle) when a coin is dragged over them, making it harder to confirm which tile will receive the drop.

## Touch/mobile inaccessibility

The component relies entirely on `onDragStart`, `onDragOver`, and `onDrop` — the HTML5 Drag and Drop API. Mobile browsers (iOS Safari, Chrome Android) do not dispatch these events for touch interactions. The game board already uses pointer events (`handlePointerDown`, `handlePointerMove`) for its pan gesture, showing the codebase is aware of the distinction. The bidding overlay has no equivalent touch path, making the entire battle phase unusable on mobile.

A minimal fix would add `onTouchStart`/`onTouchMove`/`onTouchEnd` handlers that replicate the drag logic by tracking the touch position and updating `coinPositions` on release over a tactic tile's bounding rect. Alternatively, a library like `@dnd-kit` or the `react-dnd` touch backend handles this transparently.

## Online-mode overlay positioning

In hotseat mode, the overlay component is the sole return value of `BattlePanel`, so it becomes the top-level rendered element — the `position: fixed` CSS works as a true full-screen modal. In online mode the same component is rendered *inside* a `<div className="battle-panel">` that also contains the combatants list above it:

```
<div className="battle-panel">
  <h3>...</h3>
  <div className="battle-info">combatants...</div>
  {isPart && !hasBid && <BattleBiddingOverlay ... />}  ← fixed overlay inside scrollable content
</div>
```

A `position: fixed` element inside a scrollable or transformed ancestor can behave unexpectedly in some browsers (any ancestor with `transform`, `filter`, or `will-change` creates a new containing block). The combatants section renders underneath the overlay with no interaction path — effectively invisible dead DOM. Using a React portal to mount the overlay at `document.body` would solve both concerns cleanly.

## Zero-bid confirmation

`handleConfirm` counts coins per tactic from `coinPositions` and calls `onConfirm` unconditionally. If the player never drags any coins, every tactic gets 0. The button label shows "Confirm Bids (0 coins)" which is technically accurate but in a game context is almost certainly a misclick. The physical metaphor of coins sitting untouched in a pool below tiles that visually invite dropping makes zero-bid more likely as an accident than with the old sliders.

## Removed combatant context

The previous hotseat bidding view included a combatants section showing each participant's force value and bid status. The new implementation returns only the overlay in hotseat mode, so the player commits bids without seeing army strengths — a functional regression from the previous UI that removes strategic information from the decision point.
