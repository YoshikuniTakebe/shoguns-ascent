# Ten bug fixes for Shogun's Ascent game logic

A batch of ten targeted fixes addressing game rules correctness, UI state management, and online mode parity. The changes span force calculation (Oni of Blood/Skulls honor comparison), log lifecycle (Kami markers and season archiving), a new pre-battle effect (Fire Dragon Incineration), a coin duplication exploit in bidding, hostage capture filtering, betray undo in online mode, Yurei placement state, province color styling, and a cosmetic size adjustment.

**Watch for:** Fire Dragon's Incineration effect is applied client-side in online mode but not on the server's step-by-step path, creating a state divergence when the server broadcasts (confirmed). The coin duplication fix relies on a reactive `useEffect` that may trigger unnecessary re-renders on every state update (likely). The season log archiving has two archive points (`finalizeCleanupAndAdvance` and `setupSeason`) which could double-archive if the guard condition is bypassed (possible).

## High-level view

The Oni of Blood/Skulls force fix corrects a self-comparison bug where a player alone in a province always satisfied the "lowest honor" condition. Filtering out the owner from the comparison set and requiring at least one other player present is applied consistently in both the display layer and the game logic layer.

The season log persistence fix addresses two problems: `finalizeCleanupAndAdvance` now archives the log before clearing it, and `setupSeason` gains a guard to prevent overwriting a valid archive with empty content. The two archive points must remain coordinated or the guard will silently mask future ordering bugs.

Fire Dragon's Incineration is a new 95-line function called from three sites: hotseat bid resolution, online bid resolution (client-side), and `resolveNextBattle` (non-step-by-step). The server's step-by-step path does not call it, so in online mode `determineTacticWinners` on the server operates on pre-Incineration state. The function also auto-selects which figure to kill rather than letting the player choose, diverging from the game rules.

The coin duplication fix wraps all position updates through `safeSetCoinPositions` with a key-count invariant and a phantom-coin guard. The phantom-coin guard (`prev[coinId] === undefined`) addresses the root cause: drag events firing for coin IDs not in current state.

The betray undo fix adds a `betrayUndoSnapshot` to the server lobby, captured before the first betray selection and cleared on completion. The `UNDO_BETRAY` action validates the sender is the current player before restoring.

1. **Fire Dragon not applied server-side in step-by-step battles** — The server's `SUBMIT_WAR_BIDS` handler calls `determineTacticWinners` without first calling `applyFireDragonEffect`. In online mode the server broadcast will overwrite the client's locally-applied Fire Dragon state. Import and call `applyFireDragonEffect` on the server before `determineTacticWinners`.
2. **Coin validation useEffect triggers on every render** — The safety-net `useEffect` depends on `coinPositions` and runs equality checks on every state change. Since `coinPositions` is a new object reference on each `setCoinPositions` call, this effect fires every time. The `safeSetCoinPositions` wrapper already handles the validation; the effect is redundant and should be removed or given a stable reference check.
3. **Fire Dragon kill priority is deterministic, not player-chosen** — The rules say "you must choose and kill one of your own figures" but the implementation auto-selects the weakest (bushi > shinto > monster). This removes player agency. If intentional as a simplification, document it; otherwise it needs an interactive selection step.
4. **Double-archive risk in season transitions** — `finalizeCleanupAndAdvance` archives `logHistory[currentSeason]` then `setupSeason` is called with empty log. If any code path adds to the log between these two calls, the guard in `setupSeason` will archive again, overwriting the previous archive with partial content.
5. **Betray undo snapshot serialization fragility** — `JSON.parse(JSON.stringify(...))` handles deep copy but silently loses non-serializable values. Currently safe given the state structure, but fragile under future changes.

## Fire Dragon Incineration: server-side gap in online mode

In online mode the server independently runs battle resolution. The `SUBMIT_WAR_BIDS` handler:

```
allBidsSubmitted → determineTacticWinners(l.gameState, battle)
                   ↑ no applyFireDragonEffect call
```

The server broadcasts this state to all clients. The comment in `resolveNextBattle` says "In step-by-step mode, this is already applied by the store before determineTacticWinners" — but in online mode the *server* is the authority, and it never applies it. The client's locally-applied Fire Dragon state gets overwritten on the next `GAME_STATE` broadcast.

Fix: import and call `applyFireDragonEffect` in the server's `SUBMIT_WAR_BIDS` handler before `determineTacticWinners`.

## Fire Dragon kill selection removes player agency

The rules state "you must choose and kill one of your own figures." The implementation auto-selects using a fixed priority (bushi first, then shinto, then non-immune monsters). This is a gameplay divergence — players should get to choose which figure dies. If the auto-select is intentional as a simplification, a comment explaining the trade-off would help. If not, this needs an interactive selection step similar to the hostage selection UI.

## Coin duplication: redundant reactive effect

The `safeSetCoinPositions` wrapper already validates and resets on count mismatch. The additional `useEffect`:

```typescript
useEffect(() => {
  if (Object.keys(coinPositions).length !== maxCoins) {
    const reset = {};
    for (let i = 0; i < maxCoins; i++) reset[`coin-${i}`] = 'pool';
    setCoinPositions(reset);
  }
}, [coinPositions, maxCoins]);
```

fires after every coin move because `setCoinPositions` produces a new object reference each time. It does a `Object.keys().length` check and no-ops. Since the wrapper already enforces the invariant, this effect is redundant overhead. Remove it or replace the dependency with a stable reference check (e.g., track count separately).

## Season log: two archive sites must stay coordinated

The flow:
1. `finalizeCleanupAndAdvance` — archives `logHistory[currentSeason] = [...log]`, sets `log = []`, changes `currentSeason`.
2. `doSetupSeason` → `setupSeason` checks `log.length > 0` — empty, so it skips.

If any intermediate code adds log entries between these two calls (e.g., a server broadcast), the guard fires on the *new* season name (since step 1 already changed it), overwriting the wrong key. The coupling is implicit and depends on call ordering.
