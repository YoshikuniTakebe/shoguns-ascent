# User authentication and lobby redesign for Shogun's Ascent (v2)

This change adds a full authentication system (register, login, JWT sessions), wires authenticated users into WebSocket game sessions so games persist per-user, and redesigns the games lobby into Your Turn / Waiting / Finished sections. The v2 pass addresses four issues from the v1 review: `game_players` now has a UNIQUE constraint with INSERT OR IGNORE, turn detection uses user IDs instead of username strings, the server sends AUTH_WARNING on failed WebSocket token verification, and email registration validates format. Four other v1 issues (#1 hardcoded JWT secret, #2 no token refresh, #3 token in URL, #8 unauthenticated game persistence) were deliberately accepted as appropriate for a self-hosted board game.

**Watch for:** AUTH_WARNING message is sent by the server but never consumed by the client (confirmed), player enrichment in `formatGame` matches by clanId which could produce wrong associations in edge cases (likely), snapshot volume grows unbounded with no pruning mechanism (confirmed).

**Verdict**: APPROVED

## High-level view

The AUTH_WARNING message added for v1 issue #6 is a server-side half: the WebSocket handler sends `{ type: 'AUTH_WARNING', ... }` when token verification fails, but the client's `ws.onmessage` switch statement has no `case 'AUTH_WARNING'` branch, so the message is silently dropped. The user connects as a guest with no feedback that their session isn't authenticated.

The `formatGame` enrichment logic joins `basePlayers` to `game_players` by matching `clanId`. This assumes a 1:1 user-to-clan mapping per game. For hotseat games without `game_players` entries the null fallback is correct, but if any future path allows clan reassignment within a game, the join would associate the wrong userId.

Every `broadcastState` call writes a snapshot row unconditionally. A long multiplayer game will accumulate hundreds of rows, and the replay endpoint returns all of them in a single unbounded query.

1. **AUTH_WARNING not handled on client** — The server sends `AUTH_WARNING` on WebSocket auth failure, but the client's message switch has no case for it. Add a handler that surfaces the warning to the user or triggers re-auth. (confirmed)
2. **Player enrichment matches by clanId** — `formatGame` joins players to `game_players` by `clan_id`, which assumes a 1:1 mapping per game. If two users somehow share a clanId in the same game (shouldn't happen with current lobby logic), the first match wins. Non-blocking but worth documenting the assumption. (likely)
3. **Unbounded snapshot growth** — Every `broadcastState` call writes a new snapshot row. A long game could accumulate thousands of snapshots with no pruning or compaction. Not urgent for a self-hosted game but will eventually bloat the SQLite file. (confirmed)

## AUTH_WARNING: sent but not received

The server sends the warning when a token was provided but failed verification:

```typescript
// src/server/index.ts line 486
if (!userId) {
  ws.send(JSON.stringify({ type: 'AUTH_WARNING', message: 'Token is invalid or expired. You are connected as a guest.' }));
}
```

The client's WebSocket message handler in `gameStore.ts` (around line 2719) switches on `d.type` with cases for `GAME_STATE`, `PLAYER_ID`, `LOBBY_JOINED`, `LOBBY_CREATED`, `LOBBY_STATE`, `GAME_START`, `LOBBY_CLOSED`, and `ERROR`. There is no `case 'AUTH_WARNING'` — the message falls through silently. The user connects as a guest with a random UUID, creates or joins games that won't associate with their account, and has no indication anything went wrong.

A minimal fix: add a case that sets a store field (or dispatches the existing `ruleViolationMessage`) so the UI can display a banner or redirect to the auth screen.

## Snapshot accumulation

Every game action triggers `broadcastState`, which calls `saveSnapshot` unconditionally. In a 5-player game of moderate length (hundreds of actions per season across multiple seasons), this creates hundreds to thousands of snapshot rows per game. The `getSnapshots` endpoint returns all of them in one query for replay. There's no compaction, no max-snapshot-count check, and no periodic pruning.
