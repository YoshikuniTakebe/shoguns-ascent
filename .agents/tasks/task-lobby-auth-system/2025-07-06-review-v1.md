# User authentication and lobby redesign for Shogun's Ascent

This change adds a full authentication system (register, login, JWT sessions), wires authenticated users into WebSocket game sessions so games persist per-user, and redesigns the games lobby to show categorized games (your turn / waiting / finished) with a user profile badge. The backend uses bcryptjs for password hashing and jsonwebtoken for session tokens, storing users and game-player associations in the existing SQLite database. The frontend stores tokens in localStorage and passes them as query parameters on WebSocket connections.

**Watch for:** JWT secret is hardcoded in source (confirmed), token transmitted in WebSocket URL query string is logged and cached by proxies (confirmed), no token expiry/refresh mechanism on the client (confirmed), `addGamePlayer` has no duplicate protection so rejoining a lobby could double-insert (likely).

## High-level view

Authentication uses a hardcoded default JWT secret with no rotation or per-environment override mechanism beyond a raw `process.env.JWT_SECRET` check. The 7-day token lifetime has no client-side refresh path, so users will hit silent auth failures a week after login with no recovery UX other than being bounced to the auth screen on next `checkAuth`.

WebSocket authentication passes the JWT as a URL query parameter (`?token=xxx`). The server gracefully degrades to anonymous if the token is missing or invalid, which means a network glitch stripping the query string — or an expired token — causes silent de-authentication. Games created in this anonymous state won't associate with the user account, and the only symptom is their absence from "My Games."

Game-player persistence calls `addGamePlayer` inside `startLobbyGame` for each participant, but there's no UNIQUE constraint on `(game_id, user_id)` and no INSERT OR IGNORE guard. A race between auto-start on full lobby and a manual START_GAME message could produce duplicate rows that inflate the lobby list.

The lobby UI determines "your turn" by matching `authUser.username` against `game.players[currentPlayerIndex].name`. If a user changes their display name in a future feature, or if usernames collide across accounts, the matching breaks silently and every game lands in the "waiting" bucket. Matching by user ID would be robust.

The game persistence endpoints (`/api/games/save-hotseat`, `/api/games/:id/snapshot`) and lobby CRUD have no authentication. Any client can save or overwrite game state from any origin.

## Hardcoded JWT secret and token lifecycle

The secret in `src/server/auth.ts` defaults to `'shoguns-ascent-secret-key-change-in-production'`:

```typescript
const JWT_SECRET = process.env.JWT_SECRET || 'shoguns-ascent-secret-key-change-in-production';
```

Any deployment that doesn't set `JWT_SECRET` ships with a secret visible in the repo. The token has a 7-day expiry with no refresh endpoint. The client's `checkAuth` calls `/api/auth/me` which validates the token but never issues a new one — once the token expires, the user is silently logged out with no prompt until the next page load triggers `checkAuth`. Adding a `/api/auth/refresh` endpoint or re-issuing on authenticated requests would prevent silent session loss.

## Missing uniqueness constraint on game_players

The `game_players` table schema:

```sql
CREATE TABLE IF NOT EXISTS game_players (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  game_id TEXT REFERENCES games(id),
  user_id TEXT REFERENCES users(id),
  clan_id TEXT,
  joined_at TEXT
);
```

`addGamePlayer` is called unconditionally in `startLobbyGame`. A race between auto-start on full lobby and a manual START_GAME message from the host would double-insert. `getGamesByUserId` uses an INNER JOIN without DISTINCT, so the game would appear multiple times in the lobby list. Add `UNIQUE(game_id, user_id)` to the schema or use `INSERT OR IGNORE`.

## Turn detection relies on username string matching

In `GamesLobby.tsx`:

```typescript
const currentPlayerName = game.players[game.currentPlayerIndex].name;
if (currentPlayerName === username) {
  yourTurn.push(game);
}
```

The `formatGame` server function returns `players` as the raw `players_json` array containing `{ name, clanId }` but no user IDs. The client has no way to match by ID. The server should either include user IDs in the players array or add a `currentPlayerId` field to the response.

## Silent anonymous WebSocket fallback

When token verification fails, the connection handler falls through silently:

```typescript
try {
  const reqUrl = new URL(req.url || '', `http://${req.headers.host || 'localhost'}`);
  const token = reqUrl.searchParams.get('token');
  if (token) {
    const payload = verifyToken(token);
    if (payload) { /* authenticated path */ }
  }
} catch { /* silent fallback to anonymous */ }
```

The player gets a random UUID as their ID. Any lobby they create or join won't call `addGamePlayer` with their real user ID, so the game vanishes from "My Games." Sending `{ type: 'AUTH_FAILED' }` on the WebSocket would let the client detect this and prompt re-login.

## Unauthenticated game persistence endpoints

`POST /api/games/save-hotseat` and `PUT /api/games/:id/snapshot` accept writes from any origin with no token check. While hotseat mode doesn't require auth conceptually, these endpoints allow overwriting any game's snapshot data — including online games persisted with `saveSnapshot` during play. A malicious client could corrupt an in-progress game by pushing a crafted snapshot.

## Registration email validation

The register endpoint checks `!email` (truthiness) but not format:

```typescript
if (!email || !username || !password) {
  res.status(400).json({ error: 'Email, username, and password are required' });
  return;
}
```

A user can register with `email: "x"`. This prevents any future email-based recovery and makes the uniqueness constraint on email trivially bypassable with garbage values.
