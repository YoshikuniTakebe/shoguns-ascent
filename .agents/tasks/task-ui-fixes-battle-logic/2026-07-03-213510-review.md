# UI polish, battle result layout, force display fix, and Fukurokuju fortress logic

A batch of four changes spanning visual polish, layout rework, a game logic bug fix, and a new game mechanic. The UI work reshapes the battle result popup (horizontal bets, 3-column casualties, merged poets line, larger seppuku icons), adjusts diorama figure sizing, replaces Fujin's text undo button with a circular icon, removes the Sol clan glow effect, and fixes minor layout issues in the tea ceremony and betray flows. The logic changes fix stale force values shown in step-by-step battle mode and extend the Fukurokuju monster to count as a fortress for recruit placement and province validation.

**Watch for:** The `preResolutionForces` fix (confirmed) only addresses the step-by-step path where `resData` is already present, leaving the comment "BEFORE any seppuku/hostage removals" misleading since in step-by-step mode the province already reflects those removals. The Fukurokuju fortress logic is applied consistently across all six `hasFortress` checks, but there's no coverage for what happens if Fukurokuju is killed mid-battle and a recruit phase follows in the same turn — the check relies on the figure still being present in the province.

## High-level view

The battle result popup received a layout overhaul: bets flow horizontally, casualties use a CSS grid, the Imperial Poets death breakdown merges into a single sentence, and the seppuku result section swaps generic `FigureTypeIcon` for type-specific icons (`BushiIcon`, `DaimyoIcon`, `MonsterIcon`) rendered at double size in the clan's color.

The force display bug was a data freshness issue: in step-by-step mode, seppuku/hostage/ronin changes happen via store actions before `resolveNextBattle` runs, so the province already reflects the final state. But `resData` (from `determineTacticWinners`) carried stale `participantForces` from before tactics were applied. The fix overrides `participantForces` with `preResolutionForces` in the step-by-step assignment blocks. The variable name is now a misnomer — it computes *post-resolution* forces in step-by-step mode — but the values are correct.

Fukurokuju is treated as a fortress in six places: the placement count function, the placement validation function, the store-side validation, the Luna clan helper, and two RegionCard highlighting checks (recruit target and monster target). The pattern is identical everywhere: `f.type === 'monster' && f.monsterCardId === 'sp-fukurokuju'`. Fukurokuju only counts while it physically exists as a figure in the province — killing it removes the fortress benefit immediately.

The Fujin overlay now pulls the winning player's identity from `currentTemple.winnerId` and renders a `ClanShield` with the player name in clan color. The undo button uses `UndoIcon` in a 36px circular button matching the existing Raijin pattern.

## Force display fix via `participantForces` override

`determineTacticWinners` (called when bids are submitted) stores `participantForces` in `resData` based on the province state *at that moment* — before seppuku kills figures, before a hostage is taken, before ronin are added. In non-step-by-step mode this doesn't matter because `resolveNextBattle` computes everything fresh. In step-by-step mode, the store actions mutate the province first, then `resolveNextBattle` runs with the already-mutated province. The fix adds `participantForces: preResolutionForces` to the spread of `...resData` in both step-by-step blocks (with and without Imperial Poets), ensuring the stale value from `resData` is overwritten.

The `preResolutionForces` variable is computed from `province` (the current province state at the start of `resolveNextBattle`). In step-by-step mode, this state already includes seppuku casualties removed and hostage taken. The code also explicitly adds ronin force for the hire-ronin winner by looking at the player's `ronin` count and koi clan bonus.

The comment at line 2049 ("Compute participant forces BEFORE any seppuku/hostage removals") is now misleading for the step-by-step path. It's accurate for the non-step-by-step path (where this function handles everything), but in step-by-step mode those removals already happened.

## Fukurokuju as fortress — consistency and edge cases

The Fukurokuju check is applied in six locations:

1. `calculateRecruitPlacements` — counts toward total available placements
2. `recruitPlaceFigure` — validates the target province
3. `gameStore.doRecruitPlaceFigure` — store-side validation mirror
4. `lunaHasValidProvince` — Luna clan's 2-figure-per-province limit check
5. `RegionCard` monster placement target — UI highlighting
6. `RegionCard` recruit target — UI highlighting

All six use the same ownership check (`f.owner === playerId`). The take-hostage and betray exclusions were already in place before this change.

`calculateRecruitPlacements` counts *all* matching figures across *all* provinces. If a player has both a fortress and Fukurokuju in the same province, the count is 2 from that province — Fukurokuju effectively grants +1 to the global placement budget regardless of co-location with a fortress. This matches how a second fortress would behave, but is worth confirming as intended design since the feature spec says "counts as fortress for recruit placement count" (singular).
