# Fixed-resolution map canvas with click-and-drag panning

The game map has been pulled out of responsive/percentage-based positioning and rendered instead as a fixed 1200x900px canvas inside a clipped viewport. All map elements — the background image, SVG route lines, and region cards — now live inside this fixed canvas and move together via a CSS `translate()` transform when the player click-drags. The approach eliminates layout inconsistency across screen sizes by making the map's internal coordinate space identical for everyone.

Watch for: No touch/mobile support for panning — the map is completely static on tablets and touchscreen devices (confirmed). The panning bounds allow the map to be dragged so far off-screen that 70% of the viewport is empty background (likely). Initial view pins to the top-left corner, hiding southern provinces on load (confirmed).

## High-level view

The structural change is a viewport + canvas split: `.map-container` becomes a clipped overflow-hidden viewport with `cursor: grab`, and a new `.map-canvas` child (1200x900px, `position: relative`) holds everything that should pan together. Region cards switch from percentage-based to absolute pixel positioning inside the canvas, and the SVG viewBox scales up from 800x600 to 1200x900 with coordinates multiplied by 1.5.

Panning is implemented with mouse-event handlers on the viewport div, storing translate offsets in React state. A 5px dead zone discriminates clicks from drags, protecting RegionCard click interactions. The bounds calculation allows panning up to 30% of the container's dimensions beyond the map edges in every direction — on a typical monitor with both side panels open, the center panel is roughly 700px wide, meaning the map can be dragged about 210px past its edge into empty space.

HonorTrack and AllianceDisplay sit as siblings to `.map-canvas` (outside the panning transform), so they stay fixed in the viewport. The z-index stacking means they can occlude region cards at certain pan positions with no visual cue.

## Panning implementation and input gaps

The `handleMouseMove` callback reads `containerRef.current.clientWidth/clientHeight` on every move event, triggering layout reads during drag. On complex DOMs this can cause jank. Caching the container dimensions on mousedown (they won't change mid-drag) would eliminate forced reflows.

The complete absence of touch/pointer event support means mobile and tablet users cannot pan at all. The simplest fix is switching from `onMouse*` to `onPointer*` events and adding `touch-action: none` to `.map-container`, which provides unified desktop+touch behavior in a single handler set.

## Pan bounds and initial view

The bounding formula:

```
minX = -(MAP_WIDTH - containerWidth * 0.3)
maxX = containerWidth * 0.3
```

On a ~700px-wide center panel (common at 1440p with both side panels), `maxX` = 210px — 210px of empty background visible before the map starts. Users may lose orientation when the map is mostly off-screen.

The initial translate (0, 0) pins the top-left corner of the 1200x900 canvas to the viewport's top-left. The Kyushu and Shikoku regions start hidden. A centered default:

```ts
const initialX = Math.min(0, (containerWidth - MAP_WIDTH) / 2);
const initialY = Math.min(0, (containerHeight - MAP_HEIGHT) / 2);
```

## CSS redundancy and cursor edge case

`.japan-map-background`, `.japan-map`, and `.regions-overlay` all hardcode `width: 1200px; height: 900px` despite being `position: absolute; top: 0; left: 0` children of `.map-canvas` (which is itself 1200x900). They'd fill the containing block with `width: 100%; height: 100%`. The current approach means four rules need updating if the resolution ever changes.

The `cursor: grabbing` state won't persist if the user drags fast enough that the mouse leaves the container — `handleMouseLeave` sets `isDragging = false`, snapping back to `cursor: grab` and stopping the pan. Rapid panning at viewport edges will stutter as the user repeatedly exits and re-enters the container.
