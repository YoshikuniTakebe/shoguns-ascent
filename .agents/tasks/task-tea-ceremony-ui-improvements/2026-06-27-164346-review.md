# UI polish, bug fixes, and Zorro manual placement

This change bundles seven features into a single branch: thematic SVG icons for the politics track phases, a marshal-move region glow animation, three bug fixes (recruit turn popup, seppuku honor advancement, rule violation feedback), force display in region/temple modals, battle panel styling overhaul with clan colors and seals, manual Zorro bushi placement at war start, and colored province names in the game log. The approach is additive UI/UX — no existing behavior is removed, no data model migrations needed. The only game-logic behavioral changes are the Zorro placement mode (converting automatic placement to interactive) and the seppuku honor loop fix.

**Watch for:** TemplePanel prayer force calculation is hardcoded (Luna=2x) rather than using the `calculateForce` utility, creating a divergence risk when card upgrades affect temple kami resolution (confirmed). Zorro placement does not update `activeBattles` participant lists after bushi are placed into provinces, meaning newly placed bushi may not participate in battle bidding if the battle was already constructed (confirmed). Rule violation error messages are hardcoded in Spanish with no i18n (confirmed).

## High-level view

The Zorro placement rework converts a fire-and-forget loop in `initiateWarPhase` into an interactive state machine gated by three new `GameState` fields. The `detectWarTransition` helper blocks battle start until Zorro finishes, and both placement and skip actions transition into `battleStepPhase: 'popup'`. The key concern is that `activeBattles` are constructed *before* Zorro places bushi, so those placed bushi may exist in the province but not in the battle's participant list depending on how `resolveNextBattle` reads current province state.

The temple/kami modal force display hardcodes `clanId === 'luna' ? count * 2 : count * 1` rather than calling `calculateForce`. Since temple kami resolution in `gameLogic.ts` (line ~915) uses `calculateForce` with card upgrades and honor bonuses, the modal will show wrong values whenever a player holds force-modifying season cards. The RegionDetailModal, by contrast, correctly calls `calculateForce`.

Province colors are duplicated in three locations (`BattlePanel.tsx`, `GameLog.tsx`, and `HARVEST_BADGE_META` in `GameBoard.tsx`) with no shared constant — any future province addition requires updating three files.

## Zorro manual placement state machine

The old code in `initiateWarPhase` looped through *all* provinces (not just battle provinces) and placed one bushi per empty province. The new code scopes eligibility to `warProvinceSlots` battle provinces only — a strategic improvement since Zorro's power now matters for contested provinces rather than flooding empty ones.

The concern: `activeBattles` are built by looping through war province slots *before* the function returns with `zorroPlacementActive = true`. Each `Battle` object captures `participants` (player IDs with figures in that province at construction time). If `resolveNextBattle` relies on `battle.participants` to determine who fights, Zorro's newly placed bushi will exist in `provinces[x].figures` but Zorro won't be in the battle's participant array. The resolution code at line ~1851 reads `finalProvince.figures` for force calculation, which would pick up the new bushi — but whether Zorro is included in the bidding phase depends on participant membership.

```
activeBattles constructed → zorroPlacementActive = true → function returns
                            ↓
              Zorro places bushi in province X
                            ↓
              battles[X].participants does NOT include Zorro
```

If the battle bidding only iterates `battle.participants`, Zorro won't be prompted to bid in provinces they infiltrated.

## Rule violation feedback mechanism

The approach detects failure via reference equality (`ns === gameState`), which works because `recruitPlaceFigure` returns the input state unchanged on validation failure. The store then heuristically reconstructs the failure reason by re-checking conditions. This creates a coupling risk: if `recruitPlaceFigure` adds new validation (e.g., a season-of-war restriction), the heuristic won't know about it and will show the generic fallback message. A cleaner pattern would be for `recruitPlaceFigure` to return a result type with an error code, but that would be a larger refactor.

The `isDragonfly` branch contains a nested `else if (player.clanId === 'luna')` check that is unreachable — if `isDragonfly` is true, the player's clan is `'libelula'`, so they can never also be `'luna'`. This is dead code that won't cause bugs but confuses readers.

## Duplicated province color maps

Three independent definitions of the same 8-province color mapping:

- `REGION_COLORS` in `BattlePanel.tsx` (keys lowercase)
- `PROVINCE_COLORS` in `GameLog.tsx` (keys lowercase)
- `HARVEST_BADGE_META` in `GameBoard.tsx` (different structure, has hex colors)

A refactor into `src/constants/regionColors.ts` or similar would prevent drift.
