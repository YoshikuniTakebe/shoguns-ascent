# Visual polish pass on map cards, routes, and region UI

A batch of cosmetic enhancements touching the board map's region cards, sea routes, harvest hexagons, the region diorama panel, and the Kami track. The approach is purely presentational: no state logic, no new components, no data model changes. Every modification targets readability or visual weight — color-coding region names, thickening routes for visibility, constraining icon layout, and adding texture to card backgrounds.

Watch for: The region-card transform anchor changed from center-Y to top-edge (`translate(-50%, 0)` instead of `translate(-50%, -50%)`), which shifts every card's rendered position downward by half its height relative to where it was before — the `style` prop coordinates haven't changed, so all cards will visually appear lower on the map than intended unless the position data already accounted for this (possible).

## High-level view

The region cards received the largest change density: parchment gradient background, province-colored names with white text-outline, a new `.war-token` CSS rule for gold bold battle labels, a 5-column grid lock on force indicators, and the vertical anchor flip from center to top. The anchor flip is the only change that affects layout geometry and could cause visual regression on the map without coordinate adjustments.

Sea routes switch from a single blue dashed path to a two-path stack — a wider white stroke underneath and a narrower blue stroke on top — giving the appearance of a bordered dashed line. Harvest connecting lines doubled in width (1.5px to 3px), and the harvest hexagons pick up a gold `drop-shadow` filter since `clip-path` prevents normal `border` from rendering.

The region diorama panel gets two subtle changes: layer toggle buttons switch from a gold-tinted border to solid white, and the player name in the force bar badge gains a white text-outline for legibility against dark backgrounds.

Kami track Shinto icons grow from 18px to 24px (a 33% linear increase, matching the "30% bigger" intent).

## Region card anchor shift and parchment background

The card's CSS `transform` changed from `translate(-50%, -50%)` (center-origin) to `translate(-50%, 0)` (top-left-of-center-x origin). The hover state was correctly updated to match. This means each card's `top` style value now refers to where the card's top edge sits, not its center. If the game's region position data was authored assuming center placement (which the old CSS implied), every card is now displaced downward by roughly 20-30px (half a typical card height), potentially overlapping harvest hexagons or connecting lines that sit below them.

The background shifts from the flat dark-blue `var(--bg-panel)` to a warm parchment gradient (`linear-gradient(145deg, #d4b896, #c4a882, #b89b72)`). This is a simple three-stop gradient rather than a noise-based texture — it reads as parchment-colored but lacks the grain/fiber detail of a true parchment texture.

## Province-colored region names

The `.region-name` div receives an inline `style` with color from `PROVINCE_COLORS[regionId]` (8 provinces defined). The CSS rule `.region-name { color: var(--accent-cream) }` is now dead code for all defined provinces since inline specificity wins.

Kyushu's `#F5D020` (bright yellow) on the parchment-tan background with white outline may have insufficient contrast — worth a visual check.

## Force icons grid and card width

`.region-forces` changed from `display: flex; flex-wrap: wrap` to `display: grid; grid-template-columns: repeat(5, 20px)`. This hard-caps at 5 per row, but the fixed 5-column layout allocates 100px + gaps regardless of actual troop count. Cards with 1-2 figures will be wider than necessary. Consider `repeat(auto-fill, 20px)` with `max-width` if tighter packing is desired for small garrisons.

## Harvest hex gold glow vs. border

Because `.harvest-hex` uses `clip-path`, CSS `border` would be invisible. The implementation uses `filter: drop-shadow(0 0 3px #c8a951) drop-shadow(0 0 1px #c8a951)` — two stacked drop-shadows that create a gold glow outside the clip boundary. The 3px spread with 0 offset produces a soft halo rather than a crisp border edge. An alternative approach for a sharper border: use a slightly larger background element (pseudo-element) behind the hex without clip-path, or an SVG wrapper.

## Diorama layer toggle state distinction

Both `.layer-toggle-btn` and `.layer-toggle-btn.active` now have `border: 2px solid #ffffff`. The active state is differentiated only by background color and text color, not by border color. This slightly reduces the visual distinction between active and inactive toggles compared to the previous gold-highlighted-border approach.
