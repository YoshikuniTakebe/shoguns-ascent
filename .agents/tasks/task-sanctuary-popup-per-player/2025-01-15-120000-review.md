# Per-player sanctuary popup during Kami resolution

The sanctuary reward popup is now scoped to the winning player in online mode. A new `kamiResolutionCurrentPlayerId` field tracks whose turn it is. The popup shows the reward UI only to that player; everyone else sees a "waiting for X" message. When the winner acknowledges (or completes an interactive kami ability), the server advances to the next temple and broadcasts the updated state with the next winner's id. All seven kami action types (acknowledge, Fujin move/done, Raijin place/confirm, Ryujin buy/skip) route through WebSocket in online mode with server-side player validation.

**Watch for:** Player validation on the server is soft — if `kamiResolutionCurrentPlayerId` is ever null, any player can submit actions (confirmed). The Raijin placement handler doesn't validate province eligibility beyond existence (confirmed). There is no timeout or disconnect handling if the active player drops mid-resolution (likely).

## High-level view

The design mirrors the existing harvest acknowledgment pattern: a `*CurrentPlayerId` field in game state gates who sees the interactive popup, the client sends a WebSocket action instead of mutating locally, and the server validates + advances + broadcasts.

The server handlers duplicate logic that already exists in the client store (Raijin bushi creation, Fujin move counting). This is the expected pattern in this codebase — the store has the hotseat path, the server has the online path — but the duplication means divergence bugs are possible if one path gets updated without the other.

The player validation guard `if (kamiResolutionCurrentPlayerId && ...)` means a null/undefined value skips the check entirely rather than rejecting the request. In normal flow the field is always set during active resolution, but a race condition or state corruption could open a window where any player's message is accepted.

## State field edge case on no-winner temples

If a temple has no winner (`winnerId` is null/undefined), `kamiResolutionCurrentPlayerId` becomes null for that temple index. The `ACKNOWLEDGE_KAMI_REWARD` handler handles this explicitly with an auto-advance path, so a no-winner temple won't stall. But during that brief window before auto-advance fires, the player validation guard is inert on all handlers — any incoming FUJIN_MOVE or RAIJIN_PLACE message would pass the identity check. In practice this window is near-zero (the auto-advance happens in the same handler before interactive rewards are considered), but it's the structural reason the fail-open pattern exists.

## Server handler validation — fail-open pattern

All seven handlers share this guard:

```typescript
if (l.gameState.kamiResolutionCurrentPlayerId && data.playerId !== l.gameState.kamiResolutionCurrentPlayerId) return;
```

A stricter version:

```typescript
if (data.playerId !== l.gameState.kamiResolutionCurrentPlayerId) return;
```

This would reject requests when the field is null. The no-winner auto-advance in ACKNOWLEDGE_KAMI_REWARD runs before reaching this point, so tightening the check on the interactive handlers (FUJIN_MOVE, RAIJIN_PLACE, RYUJIN_BUY, etc.) would be safe without breaking the auto-advance path.

## Raijin placement — server is now authoritative

The server RAIJIN_PLACE handler checks that the province exists and the player has bushi in reserve. It does not validate province eligibility beyond existence. Since Raijin's game rule is "summon 1 Bushi in any Province," unrestricted placement may be correct. But if future rules add constraints (e.g., not in a province under siege), the server handler is the place to enforce them — the client-side check is no longer the authority.
