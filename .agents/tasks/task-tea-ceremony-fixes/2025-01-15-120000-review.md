# Tea Ceremony UI fixes and Luna temple force

Four targeted fixes: coin display in GameLog now shows icon-then-number (dropping the keyword), Harvest VP icon turns red, Raijin overlay gains a rich player-name-and-icon message, and Luna clan figures count as force 2 in temple resolution. The BattleBiddingOverlay non-rendering was investigated and found to need no code change.

**Watch for:** Raijin overlay message is hardcoded in Spanish, breaking i18n (confirmed). Ronin log pattern still uses the old "word then icon" format creating visual inconsistency with the new coin format (confirmed).

## High-level view

The coin log formatter now extracts just the number from the matched text and puts `<CoinIcon />` before it. This produces the desired "icon 2" rendering but leaves the ronin formatter untouched, where the display still shows "2 ronin [icon]" — a visual inconsistency within the same component.

The Raijin overlay replaces a localized `t('kami.resolution.raijinPlace')` call with inline Spanish text ("coloca un … Bushi en cualquier provincia"). The template-style message with player name in clan color works visually, but the i18n key still exists unused in both `en.ts` and `es.ts`, and anyone switching locale to English will still see Spanish here.

The Luna temple force fix adds a `players` parameter to `computeTempleWinner`, applies force 2 for Luna figures, and both call sites pass `newState.players`. One open question is whether Luna's "Force 2" power should also apply in battle force calculations elsewhere in the codebase.

## Coin format extraction in GameLog

The coin regex `(\d+\s*)?\b(moneda|monedas|coin|coins)\b` captures strings like "2 coins" as the full match `m`. The fix extracts the digit with `m.match(/\d+/)` and renders `<CoinIcon />{number}`. When no number precedes the keyword (bare "coins"), `number` becomes empty string and only the icon shows — worth verifying in log messages that say something like "gains coins" without a count.

The ronin pattern at line 169-171 uses the same regex structure but still renders `{m}<RoninIcon />`, producing "2 ronin [icon]". This creates a jarring asymmetry in the same log panel. If the intent is icon-first for all resource types, the ronin handler needs the same digit-extraction treatment.

## Raijin overlay i18n bypass

The old code called `t('kami.resolution.raijinPlace')`, which maps to "Place Bushi in a province" (en) and "Coloca un Bushi en una provincia" (es). The new code builds a JSX expression with hardcoded Spanish fragments:

```tsx
{' coloca un '}
<BushiIcon size={16} color={clanColor} />
{' Bushi en cualquier provincia'}
```

This abandons the i18n layer entirely. The translation keys still exist in both locale files but are now dead code. For a single-locale game this may be acceptable pragmatically, but it sets a precedent that erodes the translation system.

## Luna clan force scope

Luna's clan description states "All figures have Force 2." The fix applies this in `computeTempleWinner` (kami resolution), which is where the user's bug manifested. If the rule also extends to battle province force calculations or the region detail modal's force display, those code paths would need the same treatment. The user's report specifically mentions temples/kami, so this appears correctly scoped for now — but worth a cross-check against game rules. (confidence: possible)
