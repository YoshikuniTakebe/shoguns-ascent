# Kami name capitalization, Path of the Lion debugging, and sea route dash spacing

Three changes ship together: kami deity names now display capitalized in log messages (a `capitalize` helper wraps `kamiType`), the sea route dash pattern is relaxed from `"8,5"` to `"10,12"`, and console.log instrumentation is added at three points around the Path of the Lion card to diagnose why its +1 daimyo force isn't being observed at runtime.

**Watch for:** The Path of the Lion card logic is *unchanged* by this diff — only debug logging was added (confirmed). The user reports the +1 force bonus never applies, but the code paths in `calculateForce`, `getFigureForce`, and `resolveNextBattle` all correctly check `hasCard(cardIds, 'sp-path-of-the-lion')`. Either the card is never reaching `player.seasonCards` at runtime (purchase not completing), or there is an upstream issue outside this diff that prevents the card from appearing in the Train modal. This commit does not fix the reported bug.

## High-level view

The Path of the Lion investigation added console.log at `buySeasonCard`, `calculateForce`, and `getFigureForce`. After exhaustive trace of the logic — deck construction (`buildSeasonDeck` always includes Core group), purchase flow (`buySeasonCard` appends to `player.seasonCards`), state transitions (`cleanupSeason` and `resolveWinter` both spread `seasonCards`), force calculation, and battle resolution — the code is consistent and correct. The bug is not a logic error in these code paths; it's either a UI/flow issue preventing the card from being purchased, or a state divergence in online mode caused by the server's incomplete `BUY_CARD` handler (which doesn't advance train resolution).

Three `console.log` statements ship to production and will fire on every force calculation involving a daimyo, polluting the console in normal gameplay. These should be gated behind `import.meta.env.DEV` or removed.

## Path of the Lion: debugging without fixing

The diff adds three console.log statements:

```typescript
// In buySeasonCard (gameLogic.ts)
console.log('[buySeasonCard]', { playerId, cardId, cardName: card.name, newSeasonCards: ... });

// In calculateForce (gameLogic.ts)  
console.log('[calculateForce] daimyo check:', { playerId, cardIds: [...cardIds], hasLion: hasCard(cardIds, 'sp-path-of-the-lion'), figForce });

// In getFigureForce (RegionDetailModal.tsx)
console.log('[getFigureForce] daimyo check:', { owner: figure.owner, cardIds: [...cardIds], hasLion: hasDisplayCard(cardIds, 'sp-path-of-the-lion'), figForce });
```

The instrumented logic chain:
1. `buySeasonCard` adds the card object (from `seasonCardsDeck`) to `player.seasonCards`
2. `getPlayerSeasonCardEffects` returns `player.seasonCards` directly (no filtering)
3. `hasCard` checks `cardIds.has('sp-path-of-the-lion') || cardIds.has('sp-path-of-the-lion-2')`

The card has `id: 'sp-path-of-the-lion'`, `group: 'Core'`, `cost: 0`. Always in the deck. Always purchasable (costs 0). The force addition (`figForce += 1`) is in the correct `fig.type === 'daimyo'` branch.

Possible failure scenarios the debugging should reveal:
- The card never shows in the Train purchase modal (UI filtering issue outside this diff — possible)
- The `seasonCardsDeck` is stale or the card was already purchased by another player earlier in the train resolution
- In online mode, the server handler doesn't advance train resolution, leaving the game stuck after purchase

## Online mode train resolution gap (likely, pre-existing)

In the Zustand store hotseat path (`doBuySeasonCard`), after `buySeasonCard` returns, the code increments `trainResolutionIndex` and calls `advanceTrainResolution`. In the server handler:

```typescript
case 'BUY_CARD': {
  l.gameState = buySeasonCard(l.gameState, data.playerId, cardId);
  broadcastState(l);
  break;
}
```

No train advancement. The next player in resolution order never gets their turn. If the user is testing in online mode, the train flow stalls after the first purchase. The card gets added to `seasonCards` correctly, but the game's progression halts, which could make it appear that nothing happened.
