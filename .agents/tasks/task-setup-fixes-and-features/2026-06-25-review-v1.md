# Home provinces fix, kami zoom modal, and EN/ES internationalization

Three independent changes bundled in one set of commits: correcting the clan-to-province mapping and introducing fixed-income economics, adding a click-to-zoom detail modal on kami temple slots, and wrapping the entire UI in a key-based i18n system with English and Spanish translations.

Watch for:
- **Interpolation only replaces the first occurrence** of a placeholder — the `replace()` call in the i18n helper is not global (confirmed).
- **TemplePanel modal text ("Shinto Figures") is not translated** despite the rest of the UI going through i18n (confirmed).
- **`CLAN_INCOME` removes territorial income** entirely, turning the harvest mandate into a no-op economically (confirmed).

## High-level view

The `HOME_PROVINCES` mapping is reshuffled and `getSeasonalIncome` is replaced wholesale: income no longer derives from province control but from a static per-clan constant. This simplifies economics but also decouples the harvest mandate from territory in a way that may not be intentional for all callers.

The kami modal is a self-contained UI addition — click a filled kami slot, get a backdrop + detail card with the kami image, effect text, and per-clan figure counts. It lives entirely in `TemplePanel.tsx` and associated CSS. It works but has a hardcoded English heading that bypasses the new i18n system.

The i18n system adds a `useT()` hook backed by Zustand's language selector, persisted to localStorage. Six components are wired up. The translation files are exhaustive for the game's current UI surface. The interpolation helper uses a single `String.replace` rather than `replaceAll`, so any key with repeated placeholders would only substitute the first.

## Fixed income replaces territorial economics

`getSeasonalIncome` previously iterated provinces, summed `harvestReward` for provinces the player controlled by force majority, and returned at least 1 coin. The new version discards all of that and returns `CLAN_INCOME[player.clanId] ?? 1` — a constant defined in `game.ts`. Income values range from 2 (Luna) to 7 (Sol, Loto), creating intentional clan asymmetry.

The concern is that `distributeSeasonalIncome` (which calls `getSeasonalIncome`) runs each season, and the harvest mandate's description still says "Gain coins from provinces you control." Under the new system this is misleading: income is clan-based, not territory-based. If there's a separate harvest-mandate code path that also calls `getSeasonalIncome` to grant coins on demand, it now returns the same flat value regardless of board state.

```typescript
// Before: income = sum of harvestReward for controlled provinces
// After:
return CLAN_INCOME[player.clanId] ?? 1;
```

This is a design decision, not a bug per se, but the mandate description string (`mandate.harvest: "Gain coins from provinces you control"`) no longer matches the implementation.

## i18n interpolation uses single-replace

Both the standalone `t()` and the `useT()` hook use:

```typescript
value = value.replace(`{${k}}`, String(v));
```

`String.prototype.replace` with a string pattern only replaces the first match. If a translation value ever contains the same placeholder twice (e.g., `"{name} attacks {name}"`), the second instance would render as the literal `{name}`. None of the current keys trigger this — every key uses each placeholder at most once — so it's latent rather than active. Switching to `replaceAll` or a regex with the `g` flag would prevent future breakage at zero cost.

## TemplePanel modal bypasses i18n

The modal introduced in `TemplePanel.tsx` renders:

```tsx
<h4 className="kami-modal-figures-title">Shinto Figures</h4>
```

This is a hardcoded English string sitting inside a component that otherwise doesn't import `useT`. The rest of the UI has been converted, so this stands out as an oversight.

## Language default and localStorage cast

The store initializes language with:

```typescript
language: (localStorage.getItem('shoguns-ascent-language') as 'en' | 'es') || 'es',
```

The cast is unsafe — if localStorage contains a corrupted or unexpected value (e.g., `"fr"`), TypeScript won't catch it at runtime and the app would try to look up translations in a non-existent locale, falling back to English via the `|| translations['en'][key]` chain. Practically this means the failure mode is graceful (English fallback), but the `language` state would hold an invalid value. A minor robustness gap.

The default is `'es'` (Spanish), meaning first-time visitors without a localStorage entry see Spanish. If this is intentional for the target audience, fine. If the game is meant to default to the browser's language or to English, this needs adjustment.
