# Force token icons and color system for card effects

This change introduces a `{force}` token to the card effect rendering pipeline, assigns semantic colors to all icon tokens (honor purple, VP red, force blue, coin gold), replaces Unicode characters (&#x26C1; and &#x2694;) with SVG icon components in the card modals, and preserves a new `CrossedSwordsIcon` for future use. The i18n strings across both locales are updated to use `{force}N` tokens instead of plain text "Force N"/"Fuerza N".

**Watch for:** The `SeasonCardsMarket.tsx` component still renders force and coins as plain text, creating an inconsistency with the modals that now use icons (confirmed). The `COLOR_MAP` applies gold to the coin icon in card effect text, but `bushi` and `shinto` icons inherit `currentColor` — whether this reads well against all card backgrounds depends on the CSS context (possible).

## High-level view

The token rendering system in `renderCardEffect.tsx` gains a `COLOR_MAP` that applies explicit hex colors to four token types and leaves two (`bushi`, `shinto`) at `currentColor`. The regex is extended to capture an optional `+` prefix on amounts, handling patterns like `{force}+1`. The color assignments are now split between `COLOR_MAP` for effect text and hardcoded hex values in the modal JSX for the coin cost display — three independent copies of `#c8a951`.

The i18n migration covers 13 cards per locale. The Kami expansion effects and battle strings still use plain "Force" text, but those aren't processed by `renderCardEffect` and appear to be a separate rendering path.

`SeasonCardsMarket.tsx` wasn't touched, leaving its force badge and coin cost as plain text while the modal views now render icons for the same data.

## Token rendering pipeline and COLOR_MAP

The approach of `COLOR_MAP[iconName] || undefined` means uncolored tokens get `undefined` passed as the `color` prop, falling through to the component's default of `currentColor`. Any new token added to `ICON_MAP` without a corresponding `COLOR_MAP` entry silently inherits `currentColor` — not a bug today, but an implicit contract that isn't documented or enforced.

## Inconsistency with SeasonCardsMarket

`SeasonCardsMarket.tsx` was listed in the context file's `relevant_files` but wasn't touched. It still renders:

```tsx
<span className="card-cost">{card.cost} coins</span>
// ...
<div className="card-force">Force: {card.force}</div>
```

The card effect text in the market *does* go through `renderCardEffect` (so `{force}` tokens in effect descriptions render with the blue fist icon), but the force badge and coin cost in the card header remain raw strings. A player seeing the same card in the market vs. the modal gets different visual treatments for the same data.

## Color constant duplication

The coin gold color `#c8a951` is defined in three places: once in `COLOR_MAP` for inline effect text, and twice as literal props in the modal components (`<CoinIcon size={16} color="#c8a951" strokeWidth="2.5" />`). If the color needs to change, all three must be updated independently.

## CrossedSwordsIcon at small sizes

The icon uses five stroke lines including small guard details at y=16. At size 14 (the rendering size in modals), the guard lines span only ~2px of rendered stroke. The icon may read as a simple X-with-a-bar rather than recognizable swords at this scale (possible).
