# Kami popup for online mode, Loto mandate color hint, figure size tweaks

Three changes bundled in one commit: the kami phase popup now triggers in online multiplayer (it was previously hotseat-only), the Loto clan's hidden mandate tiles show the executed mandate's color via border and dot, and two bushi figure sizes are adjusted. The kami fix addresses a bug where, after the fifth mandate (marshal), the game would silently stall instead of showing the kami resolution popup to the winner. The Loto visual change gives players a way to track which mandate was executed even when the tile is face-down.

**Watch for:** The kami popup detection in the GAME_STATE handler will re-fire on every server broadcast while `kamiPhasePopupPending` remains true on the server, potentially re-opening the popup after the user dismisses it (confirmed). The Loto mandate color change reveals the hidden mandate type to all players — intentional per the user requirement but worth noting as a gameplay design choice.

## High-level view

The kami popup fix mirrors the hotseat detection logic (`detectKamiPopupPending`) but places it inside the online GAME_STATE WebSocket handler. Unlike hotseat mode where local state mutations are authoritative, online mode receives repeated server broadcasts that can override the client's local dismissal state, causing the popup to re-trigger.

The Loto mandate tile change replaces a hardcoded purple border (`#4a3a6a`) with `MANDATE_COLORS[mandate.type]` and adds a 10px colored dot in the bottom-right corner. The color hint is visible to all players in both hotseat and online modes, meaning opponents can identify which mandate type Loto chose.

## Kami popup re-trigger in online mode

The new detection at line 3031 of `gameStore.ts` checks the incoming server state:

```typescript
if (state.kamiPhasePopupPending && !state.kamiResolutionActive) {
  uiResets.kamiPhasePopupVisible = true;
  uiResets.kamiPendingTemples = state.kamiResolutionTemples;
}
```

When the player dismisses the popup via `dismissKamiPhasePopup`, the local gameState gets `kamiResolutionActive: true` and `kamiPhasePopupPending: false`. But the server state hasn't changed — no action is sent to the server at dismissal time. The server only advances when the player later calls `doResolveKami`, which sends `RESOLVE_KAMI`. Between dismissal and that action, any GAME_STATE broadcast from the server (triggered by another player's action, a heartbeat, or reconnection) will re-evaluate the condition against the *server's* state, find it still pending, and set the popup visible again.

The hotseat path doesn't have this problem because `detectKamiPopupPending` runs once after a local state mutation and no external source can overwrite it.

The fix is to guard against re-triggering by checking the client's local state:

```typescript
if (state.kamiPhasePopupPending && !state.kamiResolutionActive && !get().kamiPhasePopupVisible && !get().gameState?.kamiResolutionActive) {
```

This ensures the popup only fires when the client hasn't already seen and dismissed it. **Confidence: confirmed** — the code path is fully traceable: `dismissKamiPhasePopup` doesn't send any WebSocket message, and the GAME_STATE handler unconditionally evaluates the incoming state.

## Loto mandate color visibility

The color hint is visible to *all* players in both hotseat and online modes. In online mode, each client renders the hidden mandate with its true color, which means opponents can see which mandate type Loto chose. The user requirement explicitly asks for this ("para saber que mandato se ejecuto independientemente del mandato real que esta oculto"), confirming this is intentional game-design, not an information leak bug. **Confidence: confirmed.**
