# Inline SVG icons in card effect text via token parsing

This change introduces a token system (`{vp}`, `{coin}`, `{honor}`, `{bushi}`, `{shinto}`) in card effect translation strings, replacing what were previously plain-text references to game resources and unit types. A new `renderCardEffect` utility parses these tokens at render time, substituting each with an inline SVG icon at 14px. All ~90 card effect entries in both English and Spanish have been rewritten to use tokens, and three rendering components now call `renderCardEffect()` instead of rendering raw translation strings.

Watch for: The Spanish Piedad card uses literal "Shinto" instead of the `{shinto}` token, inconsistent with both the English version and every other Spanish card that references Shinto (confirmed). The `ICON_MAP` type annotation uses `typeof VPIcon` as the value type, which works only because all icon components happen to share an identical signature — a fragile assumption if any icon diverges in the future (possible).

## High-level view

The token parser in `renderCardEffect.tsx` is a regex loop that splits a string into text segments and icon spans. It correctly resets the module-scoped regex's `lastIndex` before each call, avoiding the stale-state bug that global regexes introduce.

The translation rewrite is the bulk of the diff — mechanical substitution across ~180 entries. One card (`sp-piety` in Spanish) was left with a literal "Shinto" where every other equivalent reference uses `{shinto}`, creating a visual inconsistency where that card will show the word while others show the icon.

## Missing `{shinto}` token in Spanish Piedad

The Spanish `sp-piety` entry reads:

```
'Si ganas una Ficha de Provincia en Guerra con un Shinto, ganas {honor}1 y {vp}3.'
```

The English counterpart uses `{shinto}`:

```
'If you win a War symbol with a {shinto}, gain {honor}1 and {vp}3.'
```

Every other Spanish card that references the Shinto unit uses the `{shinto}` token (`sp-jorogumo`, `sp-komainu`, `sp-path-of-the-kannushi`, `sp-path-of-the-light`, `su-hotei`, `su-sunakake-baba`, `au-oni-of-hate`, `au-way-of-the-keiri`, etc.). The literal "Shinto" will render as plain text while the user's requirement was to replace such references with the icon.

## `ICON_MAP` typing

```tsx
const ICON_MAP: Record<string, typeof VPIcon> = { ... };
```

This asserts all values are the same component type as `VPIcon`. It compiles because all five icons share the `(props: IconProps) => JSX.Element` signature. If one icon component later adds a required prop or changes its signature, TypeScript would flag the assignment but not the callsite. Using `Record<string, React.FC<IconProps>>` would be more explicit about the contract being relied on.
