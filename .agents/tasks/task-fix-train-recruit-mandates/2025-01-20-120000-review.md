# Gating TRAIN mandate resolution to per-player turn and resetting client UI modes

This change fixes two online multiplayer bugs: (1) the TRAIN mandate card-buying modal was appearing on all clients simultaneously and allowing any player to purchase, and (2) mandate-specific UI modes (recruit provinces, marshal move/fortress) persisted after mandates ended, locking the board. The approach adds server-side turn validation on BUY_CARD and SKIP_TRAIN_PURCHASE, gates the client-side modal open/close to the current resolution player via `trainResolutionIndex`, fires the turn popup only for the relevant local player, and resets client UI mode flags when the corresponding mandate becomes inactive.

Watch for: a brief client-side race where the "accept your turn" indicator reappears between sending BUY_CARD and receiving the server response (confirmed), and absence of a first-player train popup trigger when `trainMandateActive` first appears with `trainResolutionIndex === 0` (likely).

## High-level view

The server now rejects BUY_CARD and SKIP_TRAIN_PURCHASE messages from players who aren't the current `trainResolutionOrder[trainResolutionIndex]` entry. This is the authoritative fix: even if client UI fails to lock out the wrong player, the server won't mutate state.

The `PoliticsTrack.tsx` effect tracks `trainResolutionIndex` and `localPlayerId` to open the season cards modal only when it's this client's turn, and explicitly closes it otherwise. `SeasonCardsModal.tsx` adds an `isLocalPlayerTrainTurn` guard so cards aren't interactable and the skip button is hidden for non-current players.

The turn popup logic in `gameStore.ts` now distinguishes train resolution from other mandate types. Train uses `trainResolutionIndex` changes to fire the popup (rather than `currentPlayerIndex` alone), while marshal/recruit/betray continue using `currentPlayerIndex` changes.

Client UI mode resets (`recruitMode`, `moveMode`, `buildFortressMode`, `betrayMode`, `selectedFigures`, `moveFrom`) are driven by inspecting the incoming game state: when a mandate flag is false, the corresponding local flags are cleared. This fixes the stuck recruit/marshal overlay that persisted after mandate resolution ended.

## Server-side turn validation

The server silently drops rejected BUY_CARD and SKIP_TRAIN_PURCHASE messages (early-returns with no error response). The client will remain in its current state until the next legitimate broadcast. This is acceptable for preventing exploits but produces no user feedback if a legitimate race condition causes a valid player's action to be dropped momentarily.

## Train popup trigger timing

The new popup logic fires when `trainResolutionIndex` changes and the new current train player matches `localPlayerId`:

```typescript
if (prevTrainResolutionIndex !== undefined && prevTrainResolutionIndex !== null && state.trainResolutionIndex !== prevTrainResolutionIndex) {
  newTurnPopup = lpId;
} else if (prevPlayerIndex !== undefined && prevPlayerIndex !== null && state.currentPlayerIndex !== prevPlayerIndex) {
  newTurnPopup = lpId;
}
```

The fallback to `currentPlayerIndex` change handles the case where the first resolution player differs from the current player. But if the mandate chooser IS the first resolution player, `currentPlayerIndex` won't change (they're already current). In that case `prevTrainResolutionIndex` is `undefined` (no previous train state), so the first guard clause (`!== undefined`) fails. The second condition also fails. Result: the first player never gets a turn popup.

The modal still auto-opens via the PoliticsTrack effect (which reacts to `trainMandateActive` becoming true), so the player can interact. But they miss the "it's your turn" notification that other players see. An explicit `!prevGameState?.trainMandateActive && state.trainMandateActive` trigger would close this gap.

## Accept-button flicker between buy and server response

`playerConfirmed` in `SeasonCardsModal` is local React state. After `handleConfirmBuy` fires, it resets to false immediately. Since `isLocalPlayerTrainTurn` is still true (the server hasn't responded yet), `showPlayerIndicator` becomes true, re-rendering the "accept your turn" indicator. Cards remain non-interactable because `playerConfirmed && isLocalPlayerTrainTurn` is false. But a user who clicks "accept" again enters an interactable state. The server guard catches the second BUY_CARD, but the client will process the local buy flow optimistically in hotseat mode. In online mode, `doBuySeasonCard` just sends the WS message, so no local damage occurs. Still, the flicker is user-visible and confusing.
