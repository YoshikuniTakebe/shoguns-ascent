# Bugfixes and UI improvements: recruit fortress rule, battle hotseat flow, seasonal log, styling

A batch of gameplay fixes and UX improvements across the board game implementation. The recruit mandate now enforces one figure per fortress province (with Shinto exempted from slot consumption). The hotseat battle flow is unblocked by detecting war phase transitions across all mandate resolution paths and setting the `battleStepPhase` to `'popup'`. The game log now archives entries per season with clickable tabs. Smaller UI changes: train button removed, marshal steps formatted as an ordered list, betray panel restyled with clan colors, harvest gets an initial confirmation popup, and politics track player name colors fixed.

**Watch for:** The fortress enforcement only gates `bushi` placements — monster recruitment through `recruitPlaceFigure` bypasses the one-per-fortress check entirely (confirmed). The betray mandate label uses fragile string manipulation (`replace(' -  ', '')`) that will produce garbled text for most locale strings (confirmed). The war phase detection pattern is duplicated ~12 times across the store with no extraction into a helper, creating a maintenance surface where any new mandate path that calls `advancePlayer` will silently break hotseat battles if the pattern is forgotten (confirmed).

## High-level view

The recruit fortress enforcement tracks which provinces have been used for base placements via a new `recruitUsedFortressProvinces` array on game state. The logic correctly separates base placements (one per fortress, province-locked) from bonus placements (any fortress), but the guard clause only triggers for `figureType === 'bushi'`, leaving monster figures completely unregulated despite the comment claiming otherwise.

The hotseat battle fix is brute-force: every code path that calls `advancePlayer()` now checks whether the resulting state entered the war phase with unresolved battles, and if so sets `battleStepPhase: 'popup'`. The `BattlePanel` also gained a defensive fallback that treats a null `battleStepPhase` as `'popup'` in hotseat mode. The two layers are redundant by design — a single helper in the store would eliminate both the ~12-site duplication and the need for the component fallback.

The seasonal log archives entries into a `logHistory` map each time `setupSeason` runs. The `GameLog` component uses nullish coalescing for backward compatibility with saved games that lack the field.

The betray styling extracts a "mandate label" by passing an empty `{name}` to the full `betrayNotice` translation and stripping a separator with `.replace(' -  ', '')`. This depends on exact whitespace in the translation string and will break across locales or with minor translation edits.

The marshal steps formatting splits the translated string on `/\d+\.\s/` to produce `<li>` elements — works for both current locale strings but will break silently if a translation doesn't use "1. " numbering convention.

1. **Monster figures bypass fortress enforcement** — The one-per-fortress-province guard only checks `figureType === 'bushi'`. If a monster is recruited via `recruitPlaceFigure`, it can be placed at any fortress without consuming a slot. Add `|| figureType === 'monster'` to the guard or unify the logic for non-Shinto types.

3. **War phase detection duplicated ~12 times** — The `if (ns.currentPhase === 'war' && ns.activeBattles.some(b => !b.resolved))` pattern with `battleStepPhase: 'popup'` is copy-pasted in every mandate resolution path. Extract a helper (e.g., `setWithWarTransition(stateUpdates)`) to eliminate the duplication and prevent regressions when new mandates are added.
4. **Marshal steps split regex is locale-fragile** — Splitting on `/\d+\.\s/` assumes all locale strings use numbered prefixes. A translation that uses bullet points, letters, or different numbering will produce empty or malformed list items. Consider storing steps as an array in the translation object or using a delimiter that doesn't depend on formatting conventions.
5. **Log `slice(-20)` truncates long seasons** — Both current and archived logs only display the last 20 entries. For a war-heavy season with many battles, the earlier entries are lost to the user even in the archived tab. Consider removing the cap for archived seasons or adding scroll-to-load.

## Monster figures and the fortress enforcement gap

The comment at line 556 in `gameLogic.ts` says "For bushi (and monster) placements, enforce one-per-fortress-province rule," but the guard clause immediately below only enters for `figureType === 'bushi'`:

```typescript
if (figureType === 'bushi') {
  // ...enforcement logic...
}
```

Monster recruitment through `recruitPlaceFigure` will skip this entire block. The `recruitUsedFortressProvinces` array won't be updated, and multiple monsters could be placed in the same fortress province. The comment acknowledges the intent; the implementation doesn't fulfill it.

## Betray mandate label string manipulation

The betray panel extracts a "mandate label" by calling the full `betrayNotice` translation with an empty name and then stripping a separator:

```typescript
<span className="betray-mandate-label">
  {t('actions.betrayNotice', { name: '' }).replace(' -  ', '').trim()}
</span>
```

The English string is `"Betray mandate - {name} may replace up to 2 enemy figures."` — with `{name}` as empty string this becomes `"Betray mandate -  may replace up to 2 enemy figures."` The replace targets `' -  '` (space-dash-two-spaces), which matches the English output. But the Spanish string is `"Mandato Traicionar - {name} puede reemplazar hasta 2 figuras enemigas."` — with empty name this becomes `"Mandato Traicionar -  puede..."` where the separator is `" - "` followed by one space (from the empty interpolation), making it `" -  "` only by coincidence of the interpolation gap. This works by accident in both locales today but will break if the translation adds or removes a space, or if a third locale uses a different sentence structure.

## War phase transition detection

The store sets `battleStepPhase: 'popup'` after every `advancePlayer()` call that results in a war phase state. The exact same conditional appears after `doChooseMandateTile`, `doSkipTrainPurchase`, `doTrainBuyCard`, `doSkipMarshalTurn`, `doBuildFortress`, `doRecruitPlaceFigure`, `doSkipRecruitTurn`, `doBetraySelectFigure`, `doSkipBetrayTurn`, `doAcknowledgeHarvest`, and monster placement — roughly 12 sites.

The `BattlePanel` component fallback:

```typescript
const effectiveBattleStepPhase = (isHotseat && battleStepPhase === null) ? 'popup' : battleStepPhase;
```

This means the component self-heals even if the store missed a transition. A single helper in the store (`setStateWithBattleDetection`) would eliminate both the duplication and the need for the component fallback, since all paths would be guaranteed to set the phase correctly.
