# Sidebar reserves tightening, legend tooltip, and Luna no-placement fix

This change makes three independent adjustments: tighter CSS for the sidebar reserve indicators, a new hover-triggered legend tooltip showing all game icons, and an early-return path in `doBuySeasonCard` that prevents Luna from seeing the generic placement popup when no valid province exists. The i18n keys are split into two lines for the no-placement message, and the popup now names the specific monster.

**Watch for:** The legend tooltip labels are hardcoded in Spanish, bypassing the i18n system used everywhere else in the component (confirmed). The Luna province-validity logic is now duplicated in two places that can drift apart (confirmed). The user-reported Komainu shinto sidebar issue is not addressed by this PR â€” it's a pre-existing gap in a different code path (likely, needs investigation of the `doKomainuChooseMap` -> `doPlaceMonster` flow).

## High-level view

The legend button adds a `?` icon in the game header with a CSS-only hover tooltip. The tooltip's 14 icon/label rows use hardcoded Spanish strings rather than translation keys, which diverges from the `useT()` pattern used for every other user-facing string in the same file.

The Luna fix intercepts the monster purchase flow before the placement popup appears. It checks whether Luna has any province with a fortress and fewer than 2 non-fortress figures. If none exist, `player.monsters` is incremented and the "no placement" popup is shown directly. The sidebar's `computeReserveTotals` correctly derives dual-type bonuses (shinto/daimyo) from which monster cards are not deployed on the map, so the `monsters + 1` increment is sufficient â€” no separate shinto or daimyo counter needs updating. The user's reported Komainu shinto issue is not caused by this code path (Komainu is gated by its own `sp-komainu` branch and never reaches the new early-return).

## Legend tooltip hardcodes locale

The tooltip renders 14 rows directly in JSX with Spanish labels ("Fortaleza", "Monstruo", "Moneda", "PV", "Fuerza", "Primavera", etc.) while the same file imports and uses `useT()` for every other user-facing string. The `t(...)` function is already available in scope.

```tsx
<div className="legend-tooltip-row"><BushiIcon size={20} color="#fff" /><span>Bushi</span></div>
<div className="legend-tooltip-row"><FortressIcon size={20} color="#fff" /><span>Fortaleza</span></div>
```

These should use translation keys (e.g., `t('legend.fortress')`) with corresponding entries in `en.ts` and `es.ts`.

## Luna early-return duplicates confirmMonsterPlacement logic

The province-validity check added in `doBuySeasonCard`:

```typescript
const hasValidProvince = Object.values(ns.provinces).some(province => {
  const hasFortress = province.figures.some(f => f.owner === apid && f.type === 'fortress');
  if (!hasFortress) return false;
  const lunaFigures = province.figures.filter(f => f.owner === apid && f.type !== 'fortress').length;
  return lunaFigures < 2;
});
```

is identical to the check in `confirmMonsterPlacement` at line 920. The old check was triggered when the user clicked "Accept" in the placement popup â€” so Luna would first see "Select where to place" and THEN be told there's nowhere. The new code intercepts earlier, which is the correct UX fix. But the old path remains reachable when `doKomainuChooseMap` routes Komainu to `monsterPlacementPopupVisible: true`, which then calls `confirmMonsterPlacement`. Extracting the validity check to a shared helper would prevent these from drifting apart.

## Komainu shinto sidebar gap (pre-existing, not introduced by this PR)

The `computeReserveTotals` function in `PlayerPanel.tsx` computes `shintoMonstersInReserve` based on which monster cards are NOT deployed on the map. When Komainu is placed on the map via `doPlaceMonster`, it's added as a `type: 'monster'` figure â€” not a `type: 'shinto'` figure. The sidebar then correctly excludes it from shinto-in-reserve. But if the game rule is that a deployed Komainu still counts toward the player's shinto presence (as dual-type), then `computeReserveTotals` would need a `shintoMonstersDeployed` concept that counts deployed Komainu/Hotei as shintos-on-map. Currently it only tracks them as monsters. This is out of scope for this PR but explains the user's report.
