# Online multiplayer bug fixes: TRAIN monsters, Marshal buffering, Harvest sequencing

Three online-mode bugs are fixed by splitting client and server responsibilities more carefully. Monster card placement during TRAIN now runs its popup flow locally and sends a `MONSTER_PLACED` message to the server when done, rather than expecting the server to drive UI. Marshal moves are buffered client-side with undo support, then replayed on the server when the player confirms. Harvest rewards are sequenced per-player using a new `harvestCurrentPlayerId` field that gates which client sees the interactive popup.

**Watch for:** The `MONSTER_PLACED` server handler has no player-turn validation, so any connected client can send it (confirmed). The `SKIP_MARSHAL_TURN` handler trusts the client's move/fortress list without verifying the sender is the current marshal player (confirmed). The second `useEffect` in `HarvestPopup.tsx` has an empty body and accomplishes nothing (confirmed).

## High-level view

The TRAIN fix splits monster placement into two phases: the server records the card purchase and broadcasts state, then pauses train advancement until the client sends `MONSTER_PLACED` with placement coordinates. The local store duplicates the monster-type detection logic from the server so it can drive Komainu/Hotei choice popups and Luna reserve checks client-side â€” this creates a second code path for determining placement behavior that can drift from the server's logic.

Marshal buffering accumulates moves and fortress builds in Zustand arrays (`marshalPendingMoves`, `marshalPendingFortresses`) and only sends them to the server on "end turn". The server replays the buffered actions sequentially, meaning validation (adjacency, ownership) runs on the server's pre-move state. A malformed payload could diverge from the client's intermediate states without the server detecting it.

The harvest sequencing adds `harvestCurrentPlayerId` to the game state and reorders rewards by seat order. Each client checks whether it's the "current acknowledging player" and either renders the reward popups or a waiting message. The server-side guard prevents out-of-turn acknowledgments.

## MONSTER_PLACED protocol and server-side gaps

The handler lacks the player-turn guard present on `BUY_CARD`:

```typescript
// BUY_CARD has this check:
const expectedPlayer = l.gameState.trainResolutionOrder[l.gameState.trainResolutionIndex];
if (data.playerId !== expectedPlayer) return;

// MONSTER_PLACED has no equivalent â€” any player can send it
```

Since `MONSTER_PLACED` calls `advanceTrainResolution` and potentially `advancePlayer`, a spoofed message from another client would advance the game state out of sequence.

The server uses `Math.random().toString(36).substring(2, 10)` for figure IDs in the monster placement path. These IDs end up in game state broadcast to all clients. If deterministic replay is ever needed, this will be a problem.

## Marshal race window on first move

The `suppressGameStateUpdate` flag prevents server broadcasts from clobbering local pending state during the marshal turn. The condition checks `get().marshalPendingMoves.length > 0 || get().marshalPendingFortresses.length > 0`. This means the very first move is applied locally *before* `suppressGameStateUpdate` activates â€” a broadcast arriving in that narrow window between `undoMandateState` being set and the first move being recorded would overwrite local state with server state. In practice, the server only broadcasts on other-player actions during marshal, so this race window is essentially theoretical (the current player is the only one acting). Still, the asymmetry between "have undo state" and "have pending moves" as protection conditions is fragile.

The `SKIP_MARSHAL_TURN` handler trusts `data.playerId` for the `moveForces` and `buildFortress` replay without validating that the sender is `marshalResolutionOrder[marshalResolutionIndex]`.

## Harvest empty useEffect

The `useEffect` at line 26-33 of `HarvestPopup.tsx` runs on every `harvestCurrentPlayerId` change, does nothing, and its comment claims a purpose it doesn't fulfill:

```typescript
useEffect(() => {
  if (gameState?.harvestCurrentPlayerId && gameState.harvestCurrentPlayerId !== coinConfirmedForPlayer) {
    // A new player is now acknowledging - they need to see their coin popup first
    // Only reset if coinConfirmedForPlayer doesn't match the current player
  }
}, [gameState?.harvestCurrentPlayerId]);
```

The coin popup re-appears for new players only because `coinConfirmedForPlayer === currentAcknowledgingPlayerId` evaluates to false by coincidence, not by design. Either add `setCoinConfirmedForPlayer(null)` inside the condition to make the intent explicit, or remove the `useEffect` entirely.

## Client-side monster detection duplication

In `doBuyCard`, after sending `BUY_CARD` to the server, the client runs `buySeasonCard` locally to determine if the purchased card is a monster and which popup flow to trigger. This duplicates the server's branching logic â€” checking `boughtCard.cardType === 'monster'` and matching on specific card IDs (`sp-komainu`, `su-hotei`). If the card type categorization or Komainu/Hotei ID strings ever change, both code paths need updating. A server-driven `requiresPlacement` flag in the broadcast would eliminate this coupling, but that's beyond this fix scope.
