# Monster card effects and force calculation fixes

This change fixes three force-calculation bugs (Bishamon, Sacred Warrior, Phoenix in Betray) and implements the remaining 13 monster card effects across war-phase triggers, on-enter province effects, death triggers, and movement restrictions. The approach weaves each monster's logic into the existing battle resolution pipeline and movement functions, using the same immutable-state-copy pattern established by Fire Dragon.

**Watch for:** Earth Dragon and Jorogumo effects are skipped in step-by-step (online) mode because the gameStore only calls `applyFireDragonEffect` before `determineTacticWinners` — the two new pre-battle effects are gated behind `!stepByStepMode` and have no equivalent call in the store (confirmed). Daikaiju's targeting condition degenerates (`owners.length >= 2 || owners.length >= 1`) to "any province with at least one figure," which is always true once Daikaiju itself is there (confirmed). Bishamon's trigger condition considers ALL other monsters including the owner's own, which matches the user's spec but contradicts the printed card text ("opponent Monster") — design intent should be verified (likely).

## High-level view

The force calculation gains two special-case branches: Bishamon checks for any other monster figure in the province (force 1 alone, 4 with company), and Sacred Warrior counts the player's virtue cards (1 + virtue count). The card data adds `force: 1` to Sacred Warrior but leaves Bishamon's `force: 4` in place — harmless because the special case fires first, but the leftover field is misleading for tooltips or UI that reads card data directly.

Phoenix resurrection is extended to the Betray mandate path. When Betray targets a Phoenix, instead of incrementing the owner's monster reserve, it creates a new Phoenix figure in the same province and awards 1 VP. The replacement figure from the issuer is still placed normally, so the province ends up with both the Phoenix and the replacement.

Thirteen new effects slot into three integration points: `initiateWarPhase` for Sunakake-Baba (hostage), Nure-Onna (sea-route join), and Daikaiju (fortress destruction); `resolveNextBattle` for Earth Dragon (move out), Jorogumo (take control), Kitsune (6 VP to winner), Jikininki (VP per kill), Koneko (rewards/penalties on death), and Ebisu (8 coins on death); and `moveForces` via `applyMonsterEnterEffects` for Benten (push monster out), Oni of Hate (kill figures of higher-honor players), Oni of Spite (steal VP), and Oni of Plagues (block movement).

The step-by-step battle mode (online play) only applies Fire Dragon before tactic resolution. Earth Dragon and Jorogumo have no store-level call, meaning online players never see those effects. This is the most impactful gap in the change.

1. **Earth Dragon and Jorogumo skipped in online mode** — Add `applyEarthDragonEffect` and `applyJorogumoEffect` calls to the gameStore's `determineTacticWinners` flow alongside the existing `applyFireDragonEffect` call, and persist the Jorogumo capture info on `battle.resolutionData` for later reversion.

3. **Bishamon card data still has `force: 4`** — The special case overrides it, but the field is misleading for anything that reads card data directly (tooltips, UI display). Either remove it or change it to `force: 1`.
4. **Nure-Onna auto-joins without player choice** — The card says "may cross sea route." The implementation joins unconditionally. If the player should choose, this needs a decision prompt similar to seppuku acceptance.
5. **Benten destination is arbitrary** — Benten's forced move picks the first adjacent province from the data array. If the game's design intends the Benten owner to choose the destination, this needs a player prompt.
6. **On-enter effects use stale figure reference** — In marshal movement, `applyMonsterEnterEffects` is called after province figures are updated, but the `movedFig` lookup uses `currentToFigures` which may not reflect figures removed by a prior iteration's on-enter effect. Low-risk because `applyMonsterEnterEffects` re-reads `state.provinces[provinceId]` internally, but the stale array could cause confusion in edge cases.
7. **Oni of Hate does not trigger Phoenix resurrection** — The kill loop in `applyMonsterEnterEffects` returns killed figures to reserve but never checks if the killed figure was Phoenix. Currently only bushi/shinto are targeted, but the architecture inconsistently handles Jikininki triggers (called) vs Phoenix triggers (absent) for this kill path.
8. **No server-side parity for new effects** — The WebSocket server in `src/server/index.ts` only handles seppuku's Phoenix revival. None of the 13 new effects are mirrored there. If the server validates state transitions, online clients could desync.

## Earth Dragon and Jorogumo: online mode gap

In `resolveNextBattle`, both effects are guarded by `if (!stepByStepMode)`. The gameStore's `startBattleResolution` and `startBattleResolutionOnline` call `applyFireDragonEffect` on the state before passing it to `determineTacticWinners`, but there is no corresponding call to `applyEarthDragonEffect` or `applyJorogumoEffect`. This means in all online (step-by-step) battles, Earth Dragon never moves figures out and Jorogumo never captures.

The fix requires adding both calls in the store (after Fire Dragon) and threading Jorogumo's `captured` data into `battle.resolutionData` so `resolveNextBattle` can revert it post-battle. Without the revert info, the captured figure's ownership would be permanently changed.

## Daikaiju: tautological targeting

```typescript
const targetBattleProvince = battleProvinceIds.find(bpId => {
  const bp = newState.provinces[bpId];
  if (!bp) return false;
  const owners = [...new Set(bp.figures.map(f => f.owner))];
  return owners.length >= 2 || owners.length >= 1; // always true if any figure present
});
```

Every battle province has at least one figure. The second disjunct makes the condition tautological, so Daikaiju always targets the first battle province in slot order regardless of whether it's actually contested. If Daikaiju is already in that province, it still iterates `Object.keys(newState.provinces)` and may find itself first, skipping the movement but still destroying fortresses — which could be intentional if Daikaiju was already deployed via marshal, but the intent is ambiguous.

## Bishamon force semantics

The user specified "si hay otro monstruo en la misma provincia" (any other monster), and the implementation checks `f.monsterCardId !== 'su-bishamon' && f.id !== fig.id`. The card's printed text says "opponent Monster," which would restrict the trigger to enemies only. The `force: 4` field remaining on the card definition in `SUMMER_CARDS` could confuse future developers reading the data layer — it looks like Bishamon always has force 4 when in reality the special case makes it conditional.

## Nure-Onna and Benten: automated decisions

Both effects make decisions that the card text frames as player choices ("may cross," "force a player to move"). The implementation picks the first valid option automatically, removing tactical depth from both cards. If this is a deliberate simplification for the initial implementation, the card descriptions should note it; otherwise, decision prompts are needed.
