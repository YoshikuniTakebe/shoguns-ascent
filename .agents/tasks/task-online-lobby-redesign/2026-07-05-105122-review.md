# Online lobby redesign: host-configured games with real-time clan selection

The lobby system is rebuilt around a host/join split. The host creates a game specifying player count, available clans, deck configuration, and kami settings, picks their own clan, and receives a shareable lobby ID. Joining players enter the ID, land in a lobby screen showing clan badges (clan-colored border, 20px seal, name), and pick from the remaining clans in real time. When all slots are filled and all players have chosen, the server auto-starts the game. The implementation spans server message handlers (CREATE_LOBBY, JOIN_LOBBY, SELECT_CLAN), Zustand store additions (lobbyState, sendCreateLobby, sendSelectClan), and a new LobbyScreen component with clan badge UI.

**Watch for:** Host clan can desync from the available clans list if the user edits clans after selecting one (confirmed). The auto-start condition only fires within SELECT_CLAN, so the host's manual Start button can launch a game where not all joiners have clans (confirmed). No lobby cleanup on disconnect resets the host — if the host disconnects mid-lobby, remaining players are stranded with no path forward (likely).

## High-level view

The server gains three new message types forming a lobby lifecycle: CREATE_LOBBY materializes a fully-configured lobby with the host already seated, JOIN_LOBBY adds a player without a clan, and SELECT_CLAN assigns a clan with validation and triggers auto-start when the lobby is full. The auto-start path and the existing START_GAME path now duplicate the same game initialization block, and they diverge in one important way: START_GAME doesn't check whether all players have a clan assigned.

The client splits the old single "online" flow into a Create Game panel (full configuration mirroring hotseat settings) and a Join Game panel (server URL + ID). The LobbyScreen component derives selectable clans by filtering `availableClans` against `takenClanIds` and renders them as the requested badge UI. The host's clan is pre-assigned at lobby creation time and excluded from the selectable set via the `takenClanIds` filter — but the host selection dropdown defaults to `'koi'` and isn't reset when the user removes koi from available clans, enabling an invalid-clan submission.

On disconnect, the server removes the player and broadcasts the updated lobby, but doesn't distinguish host from non-host. The client's `ws.onclose` sets `ws: null` without clearing lobby state, leaving the user on a dead lobby screen.

## Host clan selection desync

The Create Game form initializes `createHostClan` to `'koi'` and renders a `<select>` filtered to `createClans.slice(0, createPc)`. When the user removes koi from the available clans toggle, the select no longer contains a koi option, but `createHostClan` retains the value `'koi'`. React's controlled `<select>` with a value not in its options produces browser-dependent behavior — some browsers show a blank, some show the first option visually while the state remains stale.

On submission, the CREATE_LOBBY message carries `clanId: 'koi'` even though koi isn't in `availableClans`. The server doesn't validate that the host's chosen clan is within the configured available set — it trusts the client payload and inserts the host player with whatever `clanId` arrives. The fix is two-fold: reset `createHostClan` to `createClans[0]` whenever the available clans array changes (via `useEffect` or inline in the toggle handler), and add server-side validation in CREATE_LOBBY that the host's `clanId` is within `availableClans`.

## Auto-start vs manual start asymmetry

The auto-start condition in SELECT_CLAN:

```typescript
const allHaveClans = l.players.length >= l.maxPlayers && l.players.every(p => p.clanId !== '');
```

This only fires inside SELECT_CLAN. But the host also sees a permanent "Start" button that sends START_GAME. That handler has no equivalent validation:

```typescript
case 'START_GAME': {
  const l = lobbies.get(data.lobbyId || currentLobbyId || '');
  if (!l || l.host !== playerId || l.players.length < 2) return;
  // proceeds to createInitialGameState with players whose clanId may be ''
```

If the host hits Start before all joiners have picked clans, the game initializes with empty `clanId` strings, which will propagate through game state as a clan without a name, color, or abilities. Either the Start button should be disabled until all players have clans, or the server should reject START_GAME if any player has `clanId === ''`.

## Disconnect and lobby lifecycle

The `ws.on('close')` handler removes the disconnected player and broadcasts the updated lobby — or deletes the lobby if empty. It doesn't distinguish host from non-host. If the host disconnects:

- The lobby retains `l.host` pointing to a player ID no longer in `l.players`.
- No player can send START_GAME because the host check (`l.host !== playerId`) fails for everyone.
- The auto-start check requires `l.players.length >= l.maxPlayers` — with the host gone, there's one fewer player than configured, so auto-start will never trigger.
- Remaining players see a lobby screen with no way to start or leave (other than refreshing).

On the client side, `ws.onclose` sets `{ ws: null }` but doesn't reset `screen`, `lobbyId`, or `lobbyState`. There's no automatic feedback that the connection dropped — the user must manually click "Back".

## Duplicated initialization logic

The game initialization block (createInitialGameState, player ID assignment, turn order, persistence, GAME_START broadcast) appears identically in SELECT_CLAN's auto-start branch and in START_GAME. If either path is updated (e.g., adding kami initialization, or fixing the deck config cast), the other must be updated in lockstep. Extracting a `startGame(lobby: Lobby)` function would eliminate this drift risk.
