# Diorama region modal with monster and castle artwork

The region detail modal is rebuilt from a grouped-by-owner tooltip layout into a depth-layered diorama that stacks figures across three visual planes: shinto/kami in the back at 60% scale, fortress/monsters in the middle at 80%, and bushi/daimyo in the front at full size. A new utility (`figureImages.ts`) maps 19 monster card IDs and 8 clan IDs to their respective PNG imports, replacing SVG icons with artwork wherever available.

**Watch for:** The monster image map only covers spring and summer seasons (confirmed) — autumn monsters and two summer monsters (`su-sunakake-baba`, `su-yurei`) will silently fall back to SVG icons. The diorama drops all tooltip and force-count information previously available to players (confirmed), removing in-game visibility into monster effects and per-player force totals.

## High-level view

The image mapping covers every clan castle and most spring/summer monsters, but the map is incomplete relative to the card pool — no autumn monsters, no Sunakake-Baba, no Yurei. Coverage gaps fail silently to SVG icons.

The modal redesign replaces a data-rich layout (owner groups, force calculations, hover tooltips with monster names and effects) with a visual diorama that prioritizes aesthetics over information density. Figures no longer carry tooltips or labels, so the modal has become a visual display rather than an informational one. The CSS layers have no responsive breakpoints, and `transform: scale()` doesn't reduce layout size, creating overflow risk when many figures share a layer.

## Image map coverage gaps

Cross-referencing `figureImages.ts` with the card definitions in `game.ts`:

Summer monsters missing: `su-sunakake-baba` and `su-yurei` — 2 of 10 summer monsters have no entry. Autumn monsters: zero coverage across 9 cards (`au-benten`, `au-daikaiju`, `au-ebisu`, `au-kitsune`, `au-oni-of-hate`, `au-oni-of-plagues`, `au-oni-of-spite`, `au-river-dragon`, `au-sacred-warrior`).

When `getMonsterImage` returns `null`, `DioramaFigure` falls through to the generic `MonsterIcon` SVG — a significant portion of the game's monsters will render as identical silhouettes with no visual distinction between them.

## Lost information density

The prior modal grouped figures by owner, displayed clan names with colored dots, calculated force totals via `calculateForce`, and showed hover tooltips with figure type names and monster card effects. All removed:

- `calculateForce` import removed; no force numbers rendered anywhere in the modal.
- `getFigureTypeName` and `getMonsterInfo` helper functions deleted.
- `useState` for tooltip removed; `DioramaFigure` renders no text.
- `figuresByOwner` grouping replaced by layer grouping — figures from different clans are mixed within the same row, distinguished only by border color.

A player opening the modal to assess combat strength before a war mandate gets no numerical information.

## Diorama layout and scaling

`transform: scale(0.6)` on the back layer doesn't reduce the element's layout size — only its visual size. If 5+ shinto figures appear in the back layer, their flex items wrap at full layout width then shrink visually, potentially pushing the mid and front layers off-screen within the `justify-content: space-between` column. Front-row figures at `iconSize={44}` with `height: iconSize * 2.2` (~97px per image) will crowd quickly on narrower viewports given the fixed `80vw` modal width.

## Fortress rendering edge case

The `DioramaFigure` component for fortress type calls `getCastleImage(ownerClanId)`. If `ownerClanId` resolves to empty string (orphaned figure, player not in state), the function returns `null`. Unlike monsters, there is no SVG fallback for fortress — the component renders nothing visible, just an empty `<div>` with a border. The old `FortressIcon` import was removed and no equivalent path exists in the current switch statement.
