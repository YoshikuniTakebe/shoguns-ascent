# Dual-type monster reserve fix, recruit popup modal, and Kami race condition guard

This change fixes three interrelated problems: monsters that belong to two troop types (Komainu as shinto+monster, Fukurokuju as daimyo+monster+fortress) were not properly excluded from reserve counts when praying at temples; the recruit selector UI was cramped in the sidebar; and a race condition in online multiplayer allowed a player to accept the Kami summary popup before other clients rendered it, breaking the phase entirely. The approach replaces the old heuristic-based "Komainu at temple" detection with explicit `monsterCardId` tracking on temple figures, moves recruit selectors into centered portals, and adds both a server-side guard and a client-side debounce to the Kami summary acceptance flow.

**Watch for:** The temple figure `monsterCardId` tracking relies on the server passing the card ID at placement time — existing games with monsters already praying will have `undefined` for that field, causing those monsters to be counted as "in reserve" (confirmed). The 500ms debounce is a UX-visible delay that all players pay for a race that only manifests in multi-window testing (confirmed). The reserve computation is duplicated verbatim across three IIFE blocks in ActionPanel (confirmed).

## High-level view

The core data model change is adding `monsterCardId?: string` to the `Temple.figures` array type. This lets every downstream consumer — PlayerPanel reserves, ActionPanel recruit buttons, TemplePanel icons — distinguish monster figures from regular shinto figures without heuristics. The old logic tried to infer Komainu's temple status by comparing card counts against `player.monsters`, which broke for any second praying monster (Hotei).

The recruit selector popups now render via `createPortal` to `document.body` using the existing `.monster-placement-popup` CSS class, shifting from inline sidebar to centered overlay. No click-outside or Escape dismiss handler is present — only the "Cancelar" button closes them.

The Kami race condition fix is a belt-and-suspenders approach: the server rejects `KAMI_SUMMARY_READY` when `kamiSummaryVisible` is false (authoritative guard), and the client blocks acceptance for 500ms after the popup first appears (debounce safety net). The server guard alone is sufficient for correctness; the debounce adds a fixed UX penalty that all players pay regardless of whether the race is possible in their environment.

## Temple figure identity and the migration gap

The `Temple.figures` type now carries an optional `monsterCardId`:

```typescript
figures: { playerId: string; figureId: string; monsterCardId?: string }[]
```

Both the client (Komainu pray handler in `gameStore.ts`) and the server (`MONSTER_PLACED` handler in `index.ts`) now pass `monsterCardId: cardId` when creating temple figures. Every reserve calculation uses this field to build a `prayingMonsterCardIds` set that excludes praying monsters from reserve counts.

The gap: any game state serialized before this change will have temple figures with `monsterCardId: undefined`. The `prayingMonsterCardIds` set only adds entries when `f.monsterCardId` is truthy, so those legacy praying monsters will appear as if they're in reserve, inflating counts. This needs either a state-upgrade pass on load or a fallback heuristic when `monsterCardId` is absent.

## Reserve computation duplication in ActionPanel

The pattern of iterating `gameState.temples` to build `prayingMonsterCardIds` appears three times in ActionPanel (once for the shinto button IIFE, once for the monster selector IIFE, once for the daimyo selector IIFE) and once in `computeReserveTotals`. Each copy is identical:

```typescript
const prayingMonsterCardIds = new Set<string>();
gameState.temples.forEach(temple => {
  temple.figures.forEach(f => {
    if (f.playerId === cp.id && f.monsterCardId) {
      prayingMonsterCardIds.add(f.monsterCardId);
    }
  });
});
```

If the logic for determining "praying" ever changes, four separate locations need updating. The ActionPanel IIFEs could call `computeReserveTotals` from PlayerPanel (or a shared utility) instead of re-deriving the same data.

## Kami debounce as fixed-penalty defense-in-depth

The client-side debounce (`kamiSummaryVisibleSince` timestamp + 500ms check in `doKamiSummaryReady`) tracks the transition of `kamiSummaryVisible` from false to true in the WebSocket state handler and prevents the local player from firing a ready signal within 500ms. The fixed 500ms penalizes all players equally — on a local network this is pure friction, and on a high-latency connection 500ms might not be enough. Since the server guard (`if (!l.gameState.kamiSummaryVisible) return;`) is authoritative and sufficient for correctness, the debounce could be removed or at minimum documented as optional UX padding rather than a correctness requirement.

## Recruit popup missing dismiss affordances

No click-outside-to-close or Escape-key handler is visible in the diff for the portal-rendered recruit selectors. The only way to dismiss is the "Cancelar" button. The existing `.monster-placement-popup` class may handle backdrop clicks via CSS `pointer-events`, but if it doesn't, keyboard-only users and accidental popup triggers have no escape hatch beyond the button.

## TemplePanel tooltip inconsistency

When a temple figure has a `monsterCardId`, TemplePanel renders a `MonsterIcon` with a native `title` tooltip containing name, force, and effect. The PlayerPanel monster reserve tooltip uses a custom portal-positioned element with styled text. The native `title` cannot be styled, has platform-dependent hover delay, and truncates long text. For visual consistency, the TemplePanel should use the same portal tooltip pattern.
