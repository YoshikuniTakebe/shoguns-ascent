# Battle UI overhaul and Zorro placement fix

Five features bundled into one diff: CSS tweaks for modals and icons, a Zorro placement logic fix (fortress no longer blocks placement), chronological battle log markers, a redesigned battle result popup (badges, bets, seppuku section, wider layout), and bidding overlay force badges. The approach is mostly additive UI work in `BattlePanel.tsx` and a focused logic correction in `gameLogic.ts` for force display and Zorro eligibility.

**Watch for:** The Zorro placement counter can go negative (confirmed) â€” cosmetic only but confusing. The pre-resolution force display includes a duplicated hire-ronin winner determination that could drift from the canonical one downstream (likely). Inline SVG in JSX for the discount coin icon bypasses the existing `CoinIcon` component (confirmed).

## High-level view

The Zorro placement fix corrects a fortress-as-troop misclassification across three files (`gameLogic.ts`, `gameStore.ts`, `RegionCard.tsx`), and removes the auto-advance-on-last-bushi behavior so the user always confirms with "Terminar". The removal shifts battle participant updates entirely to `doZorroSkipPlacement`, which works correctly but leaves a cosmetic bug where the remaining counter can decrement below zero.

The battle result popup redesign adds three new visual sections â€” bets (Apuestas), force total badges (3 per row), and a seppuku summary â€” with a wider 700px container. Force totals now include ronin for the hire-ronin winner, computed by a new block that duplicates the tiebreak logic already present downstream in the same function.

The battle log markers wrap each battle (contested and uncontested) in `--- Inicio Batalla N (Province) ---` / `--- Batalla N resuelta ---` delimiters. Uncontested battles get their markers in the store; contested battles get them in `resolveNextBattle`.

CSS and icon changes are cosmetic: modal widths, legend icon colors, shinto icon 20% larger, discount text now shows "-1" with an inline SVG, card zoom always available even when unaffordable.

## Zorro placement: negative counter on exhausted allotment

With auto-advance removed, `doZorroPlaceBushi` no longer checks `zorroPlacementsRemaining <= 0` before allowing placement. The `bushi <= 0` guard and the `hasOwnFigure` guard prevent invalid placements in practice (since `remaining` was initialized to `min(bushi, eligibleProvinces.length)`), but the displayed counter can show "-1 restantes" if the player places after exhausting their allotment. A `zorroPlacementsRemaining <= 0` early return would eliminate this.

## Pre-resolution force display and hire-ronin winner duplication

The new `preHireRoninWinner` block at the top of `resolveNextBattle` mirrors the `hireRoninWinner` block ~200 lines later almost verbatim â€” same iteration over participants, same bid comparison, same honor tiebreak. Since hire-ronin winner is determined solely by bid amounts and honor position (neither of which changes during seppuku/hostage), the two will always produce the same result today. But having the same 15-line tiebreak algorithm in two places creates drift risk: a future change to tiebreak logic that updates one but not the other would silently break force display.

The Koi clan branch in the pre-resolution block adds `remainingCoins` to force using `player.coins - totalBidByPlayer` on the pre-mutation state. Since `calculateForce` does NOT include Koi coin-as-ronin (that's a hire-ronin-specific bonus, not a base force mechanic), this is correct today. But it's an implicit coupling: if `calculateForce` ever gains Koi-coin awareness, the pre-resolution display will double-count.

## Battle log numbering divergence risk

Contested battles use `unresolvedIdx + 1` (from `activeBattles.findIndex(b => !b.resolved)`) while uncontested battles use `currentBattleIndex + 1` (same expression). Both resolve to the same value under the current sequential processing flow. If any future mechanism resolves battles out of order, the two numbering sources could produce different numbers for the same logical battle slot.

## Inline SVG bypasses CoinIcon component

Both discount displays in `SeasonCardsModal.tsx` render a raw `<svg>` with a circle and two crossing lines (visually a plus/crosshairs, not a coin). The existing `CoinIcon` component accepts `color` and `size` props and could serve this purpose. The inline approach duplicates the icon in two places in the same file and produces a different visual than `CoinIcon` renders elsewhere.
