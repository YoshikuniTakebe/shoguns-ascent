# i18n for season card effects and names

This change wires all 85 season card effects and names through the existing i18n system so they render in the active language instead of hardcoded English. A utility module (`cardTranslations.ts`) maps card IDs to translation keys, stripping the `-2` suffix from duplicate cards. Four components are updated to call `t(getCardEffectKey(card.id))` and `t(getCardNameKey(card.id))` instead of reading `card.effect` / `card.name` directly. 170 new translation keys (85 names + 85 effects) are added to both `en.ts` and `es.ts`.

**Watch for:** The `as TranslationKey` cast in `cardTranslations.ts` silences TypeScript for dynamically-constructed keys, meaning a typo or missing key will fail silently at runtime rather than at compile time (confirmed). The `getMonsterInfo` helper in `RegionDetailModal.tsx` uses the standalone `t` outside React's render cycle — currently safe because the parent re-renders on language change, but fragile if the result is ever memoized (confirmed). Spanish translations are missing diacritics throughout (confirmed).

## High-level view

The translation key derivation relies on a runtime string concatenation cast to `TranslationKey`. Any card added in the future without a corresponding translation entry will silently render the raw key string to users (e.g., `card.effect.xx-new-card`) rather than produce a compile-time error.

`RegionDetailModal` calls the standalone `t()` from a module-level function (`getMonsterInfo`). This currently works because the component subscribes to language via `useT()` and re-renders, causing `getMonsterInfo` to re-execute with fresh state. The pattern would break if anyone memoizes or caches that function's output.

The Spanish translations use ASCII approximations for accented characters (`Otono` instead of `Otoño`, `Dragon` instead of `Dragón`, `simbolo` instead of `símbolo`). This is consistent with the pre-existing `es.ts` style but affects readability for Spanish speakers.

<summary>Issues (3)</summary>

1. **Unsafe type cast for translation keys** — `getCardEffectKey` / `getCardNameKey` cast dynamically-built strings to `TranslationKey`. If a card is added to `game.ts` without a matching entry in `en.ts`/`es.ts`, the UI will silently display the raw key string. Consider a dev-time assertion or map validation.
2. **Fragile translation pattern in getMonsterInfo** — The module-level `getMonsterInfo` function uses `tStandalone()` (reads language via `getState()`). It works today because the caller re-renders on language change, but would break if the result were ever memoized. Move the translation into the component render path to make the contract explicit.
3. **Missing diacritics in Spanish translations** — Words like `Dragon`, `simbolo`, `Otono`, `Leon` lack accents (`Dragon` -> `Dragón`, `simbolo` -> `símbolo`, `Otono` -> `Otoño`, `Leon` -> `León`). This is consistent with the existing `es.ts` but affects user-facing quality for Spanish speakers.

## Type safety trade-off in cardTranslations.ts

```typescript
export function getCardEffectKey(cardId: string): TranslationKey {
  return `card.effect.${getBaseCardId(cardId)}` as TranslationKey;
}
```

The `as TranslationKey` cast bypasses TypeScript's literal type checking. The `TranslationKey` type is derived from `keyof typeof en`, so only keys literally present in `en.ts` are valid. A template literal type constrained to actual keys would catch mismatches at compile time, but would require refactoring the card ID type. The fallback behavior in the `t()` function returns the key itself when no translation exists (`translations[lang][key] || translations['en'][key] || key`), meaning a missing entry surfaces as the literal string `card.effect.xx-missing` in the UI with no console warning.

## Reactivity gap in RegionDetailModal

`getMonsterInfo` is declared at module scope and calls `tStandalone(getCardNameKey(card.id))`. The standalone `t` reads language from `useGameStore.getState()` — a point-in-time snapshot, not a subscription. The component subscribes to language via `useT()`, so it re-renders on language change, causing `getMonsterInfo` to re-execute with current state. The risk materializes if someone wraps the call in `useMemo` or caches the tooltip text — at that point the stale closure captures the old language. The fix is straightforward: pass the `t` function from `useT()` into `getMonsterInfo` or inline the translation in the render body.

## Diacritics in Spanish locale

The pre-existing `es.ts` already uses ASCII approximations throughout (e.g., `'legend.autumn': 'Otono'`). This change follows the same convention. Since the user's request specifically targets Spanish display quality, this is worth noting: all 85 effect translations and many name translations contain words that should carry accents in standard Spanish orthography.
