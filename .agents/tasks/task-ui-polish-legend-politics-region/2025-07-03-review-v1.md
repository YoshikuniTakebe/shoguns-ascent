# Legend tooltip, politics track phase indicators, and region modal shadow

This change adds visual polish across three areas: the legend tooltip gains an upward arrow and season-colored entries with a larger font; the politics track phase diamonds (tea, kami, war) now glow when their respective phase is active and show descriptive tooltips on hover; and the ClanShield in the region modal diorama gets a drop-shadow for better contrast. The AutumnIcon SVG was also replaced with a more recognizable momiji (maple leaf) shape.

**Watch for:** The kami diamond `active` class is driven by `kamiResolutionActive` rather than `currentPhase`, which means it only glows during kami reward resolution—not during the entire kami "turn" between mandate groups (confirmed). The hardcoded Spanish tooltip strings bypass the i18n system already used elsewhere in this codebase (confirmed).

## High-level view

The legend tooltip arrow uses `::before` with `border-bottom-color` to point upward, matching the existing tooltip arrow pattern but inverted. Font size goes from 0.75rem to 0.85rem. Season rows pass each season's color to both the icon and an inline `style` on the text span, meaning the color palette lives in JSX rather than CSS variables.

The politics track phase diamonds gain a conditional `active` CSS class that triggers `pulse-gold` (tea, kami) or a new `pulse-red` (war) keyframe animation. Each diamond is wrapped in a `.politics-phase-wrapper` div hosting a hover tooltip. The tooltip text is hardcoded in Spanish: "Ceremonia del Te", "Turno Kami", "Fase de Batallas"—bypassing the `useT()` hook used elsewhere.

The ClanShield drop-shadow is applied via an inline `filter` on the parent div, repeated across five code paths in `DioramaFigure`. A CSS class would reduce repetition but isn't blocking.

## Kami diamond activation semantics

The tea and war diamonds derive their `active` class from `gameState.currentPhase === 'tea'` and `gameState.currentPhase === 'war'` respectively. The kami diamonds use `gameState.kamiResolutionActive`. Tracing through `gameLogic.ts`, `kamiResolutionActive` flips to `true` only when a kami temple is actively paying out rewards (around line 1448) and resets to `false` once all temples have been resolved (around line 1522). Between mandate slots 2→3, 4→5, and 6→7 there is a logical "kami turn" in the politics flow, but during the popup-pending stage before resolution starts, the diamond won't glow.

If the design intent is "glow whenever the game is processing kami stuff," the condition should also check `gameState.kamiPhasePopupPending`. If the intent is only "glow during active resolution," the current implementation is correct but may feel inconsistent with tea/war which glow for their entire phase.
